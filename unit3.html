Processes & Threads

🔮
2026 Prediction — VERY HIGH: "Differentiate between the process and thread" appeared in 2025 Part A Q5. New anchor topic. Absent from 2023 and 2024 — brand new in 2025, confirmed to persist. Based on the RTU pattern (2026 DIP, ML: new 2025 Part A topics retained), Process vs Thread is VERY HIGH for 2026 Part A.

What is a Process?

A process is a program in execution — an independent unit of execution with its own dedicated memory space (address space), program counter, registers, stack, and operating system resources. Each process runs in complete isolation from other processes.

A process is like a person working alone in their own private office. They have their own desk, computer, files, and tools. If two people want to share information, they have to send notes (messages) between offices — they can't directly touch each other's stuff. If one person quits, the other is completely unaffected.
Process — Key Properties
Own address space (code, data, heap, stack) — private
Own process ID (PID) assigned by OS
Own file descriptors, environment variables, resources
Isolated from other processes — crash = no impact on others
Heavy to create (fork() in Unix creates full copy of address space)
IPC needed for communication: pipes, sockets, message queues, shared memory
Typical process creation time: 100μs–10ms. Context switch: 1–10μs.

What is a Thread?

A thread (also called a lightweight process) is the smallest unit of execution within a process. Multiple threads share the same process address space — code, data, and heap — but each thread has its own stack, program counter, and registers.

Threads are like multiple workers sharing ONE office. They all use the same desk, computer, and files (shared memory). If one worker needs to tell another something, they can just talk directly — no notes needed (shared variables). Very fast to communicate but risky: if one worker accidentally shreds an important file, everyone is affected. Also, if one thread crashes badly, the whole process can die.
Thread — Key Properties
Shares address space with parent process (code, data, heap) — shared
Own stack, PC, registers — private per thread
Lightweight: cheap to create, cheap context switch
Direct communication via shared variables (needs synchronization!)
Crash of one thread can kill the whole process
Typical thread creation time: 10μs–100μs (10x faster than process)

Graph Models for Process Representation

Why Graph Models?

In distributed systems, multiple processes run concurrently and interact. Graph models provide a visual and formal way to represent processes, their interactions, dependencies, and execution order. They help detect deadlocks, analyze concurrency, and reason about correctness.

Graph models are like organizational charts or flowcharts for your processes. Instead of just saying "Process A talks to Process B," you draw a box for each process and an arrow showing the direction of communication or dependency. This helps you spot problems: if arrows form a circle, you might have a deadlock!

1. Process Interaction Graph

Represents processes as nodes and communication channels as directed edges. An edge from P1 to P2 means P1 can send messages to P2.

Fig: Process Interaction Graph — P1 sends to P2 and P4; P2 sends to P3; P4 sends to P3

2. Wait-For Graph (WFG)

A special graph used for deadlock detection. Directed edge from Pi to Pj means "Pi is waiting for Pj to release a resource." A cycle in the WFG indicates a deadlock.

Wait-For Graph — Deadlock Detection
Node: Each process in the system
Edge Pi → Pj: Pi is waiting for a resource held by Pj
Deadlock exists iff there is a CYCLE in the WFG
Example: P1→P2→P3→P1 (cycle) = DEADLOCK
P1→P2→P3 (no cycle) = No deadlock
In distributed systems, no single node has the global WFG — Chandy-Misra-Haas constructs it by propagating PROBE messages.

3. Precedence Graph (Task Graph)

Used for representing the execution order of tasks. Edge from Ti to Tj means "Ti must complete before Tj begins." Used in static scheduling to assign tasks to processors.

Precedence Graph Properties
Directed Acyclic Graph (DAG) — no cycles (tasks can't depend on themselves)
Edge Ti → Tj: Ti is a prerequisite of Tj
Critical path: longest path in the DAG = minimum execution time
Used in: compiler optimization (instruction scheduling), distributed job scheduling, MapReduce DAG in Spark.
🔗
Connection: WFG → Distributed Deadlock (Unit 5)

The Wait-For Graph introduced here is the core data structure for distributed deadlock detection in Unit 5. Chandy-Misra-Haas algorithm builds the distributed WFG by sending PROBE messages. Understanding graph models here makes deadlock detection easier to understand later.

Client/Server Model

Overview

The Client/Server model is the dominant paradigm for building distributed applications. It separates the system into two asymmetric roles: clients that request services and servers that provide them. The model forms the basis of RPC and most internet applications.

The client-server model is exactly like a restaurant. You (the client) sit at a table and place an order (request). The kitchen (server) prepares your food (service) and sends it out. You never enter the kitchen — you just make requests through the waiter (protocol). The kitchen serves many customers simultaneously.

How Client-Server Communication Works

Fig: Client/Server interaction — client issues request and blocks; server executes and sends reply; client unblocks

Characteristics of Client/Server Model

Client Side Properties
Initiates communication (active role)
Sends request → waits (blocks) for reply
Typically runs on end-user machine
Doesn't need to be always running (intermittent)
Many clients can connect to one server
Server Side Properties
Waits passively for requests (reactive role)
Must be always running (daemon process)
Handles multiple clients simultaneously (threads/processes)
Has a well-known address/port that clients can find
Stateless (preferred) or stateful (complex but powerful)

Stateless vs Stateful Servers

AspectStateless ServerStateful Server

Microkernel OS Organization in Client-Server Fashion:

In a traditional monolithic OS, ALL services (file system, device drivers, networking) run in kernel space. In a microkernel, only the bare minimum runs in kernel space; everything else runs as user-space server processes.

Microkernel structure:
· Kernel (minimal): Process management, IPC, memory management, interrupt handling
· File Server (user space): Handles file system requests
· Network Server (user space): Handles networking
· Device Driver Server (user space): Handles hardware

Disadvantages: More IPC overhead than monolithic kernel (more message passing between user-space servers).

Examples: GNU Hurd, QNX, Mach (used in macOS), Minix.

Time Services in Distributed Systems

Why Time Services?

In a distributed system, each machine has its own hardware clock that drifts over time. Without synchronization, two events that should have the same timestamp will have different ones — causing ordering problems, transaction failures, and authentication issues.

Imagine every worker in your office sets their own clock. After a week, some are 5 minutes fast, some 3 minutes slow. Now when you look at when documents were created or modified, you can't trust the timestamps. Time Services are like a central clock on the wall that everyone syncs to — so all clocks show the same time.

Network Time Protocol (NTP)

NTP is the standard internet time synchronization protocol. It synchronizes all clocks on a network to within a few milliseconds of UTC (Coordinated Universal Time).

NTP — Stratum Hierarchy
Stratum 0: Atomic clocks, GPS receivers (nanosecond accuracy)
Stratum 1: Time servers directly connected to Stratum 0 (microsecond accuracy)
Stratum 2: Synchronized from Stratum 1 (millisecond accuracy)
Stratum 3: Synchronized from Stratum 2
...
Your PC: Typically Stratum 3 or 4 (accuracy: ~50ms)
NTP protocol: client sends request with T1 timestamp; server replies with T2 (receive) and T3 (send); client receives at T4. Offset = ((T2-T1)+(T3-T4))/2

Berkeley Algorithm (Internal Synchronization)

NTP synchronizes to an external time source. The Berkeley Algorithm synchronizes clocks among machines in a group without an external reference — useful in a closed network.

Berkeley Algorithm — Steps
Step 1: Time daemon (master) polls all machines for their current time
Step 2: Machines send their clock values to the master
Step 3: Master computes the AVERAGE of all clock values
(ignoring outliers that differ too much)
Step 4: Master sends each machine the DIFFERENCE it should adjust by
(not the absolute time — to avoid sudden jumps)
Step 5: Machines adjust their clocks accordingly
Key: Machines are told HOW MUCH to adjust, not WHAT TIME IT IS. Avoids clock jumps.

DCE Distributed Time Service (DTS)

DCE's DTS (covered in Unit 2) provides time synchronization as part of the DCE framework. Key feature: provides a skew bound — the guaranteed maximum difference between any two clocks in the system.

🔗
Connection: Time Services → Logical Clocks (Unit 2)

Physical time synchronization (NTP, Berkeley) tries to keep physical clocks close. But even with NTP, there's ±50ms error — too much for distributed transactions. This is EXACTLY why Lamport's logical clocks (Unit 2) were invented: we don't need real time, just event ordering.

Language Mechanisms for Synchronization

🔮
2026 Prediction — HIGH (Part C Q2): "Explain the language mechanisms for synchronization" appeared as Part C Q2 in 2025 (10 marks). Based on the RTU pattern of retaining recent Part C topics, this is HIGH confidence for 2026. Always draw the monitor and semaphore diagrams.

Why Language-Level Synchronization?

In concurrent programs, multiple processes/threads access shared resources. Without synchronization, race conditions occur — the final result depends on the timing of operations. Language mechanisms provide constructs to control access to shared resources, ensuring correctness.

Imagine a shared Google Doc that two people edit simultaneously. If both read "Hello", both add "World", and both save — only one person's change wins. Synchronization is like a rule that says "only one person can type at a time." Language mechanisms are the different tools (locks, signals, message queues) for enforcing these rules.

1. Semaphores

A semaphore is an integer variable used for mutual exclusion and synchronization, accessible only through two atomic operations: wait (P) and signal (V).

Semaphore Operations
Semaphore S initialized to some value N
wait(S) — also called P(S) or down(S):
while (S == 0) busy_wait; // or block
S = S - 1; // atomic
signal(S) — also called V(S) or up(S):
S = S + 1; // atomic
(wake up a waiting process if any)
Binary Semaphore (Mutex): S = 0 or 1 → Mutual Exclusion
Counting Semaphore: S = N → Control N concurrent accesses
Example: Printer has 3 units → semaphore S=3. Each user: wait(S) before printing, signal(S) after.
Fig: Binary semaphore for mutual exclusion — only one process can be in the critical section at a time

2. Monitors

A monitor is a high-level synchronization construct that encapsulates shared variables, procedures operating on them, and the synchronization needed. Only one process can be active in a monitor at a time — mutual exclusion is automatic.

A monitor is like a private meeting room with a key. You can only enter if no one is inside (mutual exclusion is built-in). Inside, you declare all shared data and the operations on it. If you need to wait for a condition (e.g., wait until buffer is not empty), you use a condition variable to sleep and be woken up.
Monitor Structure
monitor MonitorName {
shared variables; // All shared data here
condition c1, c2; // Condition variables for waiting
procedure P1(...) { ... } // Only one active at a time
procedure P2(...) { ... } // Mutual exclusion automatic
initialization code;
}
Condition variable operations:
c.wait() : release monitor lock, process sleeps on c
c.signal() : wake up one process sleeping on c
Monitors vs Semaphores: Monitor = object-oriented, automatic mutual exclusion. Semaphore = low-level, manual. Java synchronized keyword implements a monitor.

3. Message Passing for Synchronization

In distributed systems without shared memory, message passing is the primary synchronization mechanism. A process synchronizes with another by sending/receiving messages — the act of receiving a message implicitly synchronizes the receiver with the sender's state.

Rendezvous (ADA Programming Language)
Server process declares ENTRY points (like procedures)
Client process calls entry: client.call(Entry_Name, params)
Server accepts with: accept Entry_Name do ... end
Both BLOCK until the other is ready — rendezvous (meeting point)
After rendezvous: both continue independently
ADA's rendezvous combines synchronization + communication in one construct. CSP (Communicating Sequential Processes) uses similar channel-based communication.

4. Mutexes and Condition Variables (POSIX)

The most practical synchronization primitives in C/C++/Java concurrent programming:

Mutex Operations
pthread_mutex_lock(&mutex) // Acquire lock — blocks if held
pthread_mutex_unlock(&mutex) // Release lock
Condition Variable Operations (used WITH mutex):
pthread_cond_wait(&cond, &mutex) // Release mutex + sleep atomically
pthread_cond_signal(&cond) // Wake one sleeping thread
pthread_cond_broadcast(&cond) // Wake ALL sleeping threads

2023 Part A Q6 · 2 marks ⭐⭐Unit 3Concurrency
Concurrency in DS means multiple processes run simultaneously across different nodes, potentially accessing shared resources.

Handling mechanisms:
1. Semaphores/Mutexes: Lock shared resources before access (local within a node)
2. Distributed Locking: Centralized lock manager or token-based distributed locks
3. Timestamp Ordering: Each operation gets a global timestamp; conflicts resolved by timestamp order

5. Message Passing: No shared memory → each resource has one owner; access via messages (no race conditions possible across nodes)

Object Model Resource Servers

What is an Object Adapter? (2023 Part B Q3)

In distributed object systems (like CORBA), an Object Adapter is the component that connects the ORB (Object Request Broker) to the actual object implementation. It manages the lifetime of objects and dispatches method calls from the ORB to the correct object.

The Object Adapter is like a hotel receptionist for your objects. When a call comes in for "Room 302" (an object ID), the receptionist (object adapter) checks if that room is occupied (object is active), wakes up the guest if needed (activates the object), and directs the caller to the right room (dispatches the call). Without the receptionist, every caller would wander the hotel looking for their contact.
Object Adapter Functions
1. Object Registration: Register objects with the ORB so they can be found
2. Activation/Deactivation: Create/destroy object instances on demand (lazy activation)
3. Request Dispatching: Forward incoming method calls to the correct object + method
4. Object Identity: Assign and manage unique Object IDs (OIDs)
5. Persistence: Optionally persist object state between requests (stateful objects)
CORBA defines the Portable Object Adapter (POA) — a standardized object adapter that works across different ORB implementations.

Object Model in Distributed Systems

The distributed object model extends the local object model across a network. Objects have:

  • Interface: The set of methods that can be called remotely (defined in IDL for CORBA)
  • Object Reference: A unique identifier that clients use to invoke methods — acts like a network address for the object
  • Implementation: The actual code running on the server — hidden from the client
🔗
Connection: Object Adapter → CORBA (Unit 6)

The Object Adapter is a core CORBA component. When you study CORBA in Unit 6, the POA (Portable Object Adapter) is the standardized object adapter. Understanding the object adapter here makes CORBA architecture much clearer.

Message Passing (IPC)

🔮
2026 Prediction — HIGH (Part B Q3): "Explain Inter-Process Communication Mechanism" appeared as Part B Q3 in 2025 and Part A Q3 in 2024. TWO consecutive years. IPC mechanism (especially message passing) is a stable fixture — HIGH confidence for 2026 Part B.

What is Message Passing?

Message passing is the fundamental communication mechanism in distributed systems. Since processes on different machines cannot share memory, they communicate by explicitly sending and receiving messages over the network. It is the building block for all higher-level protocols (RPC, RMI, message queues).

Message passing is like postal mail between offices. If two offices need to share information, they write a letter (message), put it in an envelope (packet), and send it through the post office (network). The receiver opens their mailbox (socket/queue), reads the letter, and sends a reply. No one needs to physically visit the other office — all communication is through letters.

Synchronous vs Asynchronous Message Passing (2024 Part B Q2)

IPC Mechanisms — Complete Overview (2024 Part A Q3)

IPC Mechanisms in Distributed Systems
1. PIPES (within same machine):
Unidirectional byte stream between related processes (parent-child)
Anonymous pipe: ls | grep .txt (Unix)
Named pipe (FIFO): works between unrelated processes
2. MESSAGE QUEUES:
Ordered queue of messages; decoupled producer/consumer
Asynchronous; supports persistence; priority ordering
Examples: Apache Kafka, RabbitMQ, AWS SQS
3. SHARED MEMORY (same machine):
Fastest IPC — processes map same physical page into address space
Requires synchronization (semaphores) to prevent races
4. SOCKETS:
Network IPC — works across machines and OS
TCP sockets: reliable, ordered, connection-oriented
UDP sockets: unreliable, unordered, connectionless — faster
5. REMOTE PROCEDURE CALL (RPC):
Higher-level IPC — makes remote calls look like local calls
Built on top of sockets; hides all network communication
6. SIGNALS (within same machine):
Asynchronous notification to a process (e.g., Ctrl+C → SIGINT)
In distributed systems, sockets → message passing → RPC are the primary IPC mechanisms. Pipes and shared memory are local-only.

Request/Reply and Transaction Communication

Request/Reply Protocol

The request/reply pattern is the simplest distributed communication protocol. A client sends a request message; the server processes it and sends a reply message. This is the foundation of RPC.

Request/Reply Protocol — 3-Message Pattern
Standard: 2 messages — REQUEST and REPLY
With acknowledgement (for reliable delivery): 3 messages
1. Client → Server: REQUEST(operationId, params)
2. Server → Client: REPLY(result)
3. Client → Server: ACK (if using 3-message protocol)
Message structure: {messageType, requestId, objectReference, operationId, arguments}
Idempotency: If request is sent twice (retransmission), the result should be the same. GET is idempotent; POST may not be. Use requestId to detect duplicates.

Transaction Communication

A transaction is a sequence of operations that are treated as a single unit — either ALL succeed or ALL fail. Distributed transactions require coordination across multiple machines.

ACID Properties (Mandatory for Exam)
A — ATOMICITY: All operations in transaction succeed, or none do (rollback)
C — CONSISTENCY: Transaction moves database from one consistent state to another
I — ISOLATION: Concurrent transactions appear to execute serially (no dirty reads)
D — DURABILITY: Committed transaction persists even if system crashes (write-ahead log)

Two-Phase Commit (2PC) Protocol

The standard protocol for ensuring atomicity across distributed nodes in a transaction:

Fig: Two-Phase Commit. Phase 1: Coordinator sends PREPARE to all. Phase 2: If all vote YES → send COMMIT; if any NO → send ABORT.

Name and Directory Services

Why Naming?

In a distributed system, resources (files, services, processes) can be anywhere on the network. Clients need a way to find them. Naming services provide a mapping from human-readable names to network addresses — just like phone directories map names to phone numbers.

Naming is like your phone's contacts list. You don't remember your friend's phone number — you just search for "Mom" and your phone finds the number. Naming services do this for distributed resources: you say "FileServer1/reports/Q3.pdf" and the name service translates that to an IP address and path.

DNS — Domain Name System (The World's Largest Name Service)

DNS Name Resolution
Name: www.google.com → IP: 142.250.195.4
Hierarchy: com → google.com → www.google.com
Resolution: Local cache → Recursive resolver → Root server → TLD server → Authoritative server
Caching: TTL-based (reduces load — most lookups resolved from cache)
DNS is a distributed, hierarchical, replicated name service — a perfect example of a distributed system solving the naming design issue.

LDAP — Lightweight Directory Access Protocol

LDAP is a directory service protocol for looking up and managing information about network users and resources. Organized as a hierarchical tree (DIT — Directory Information Tree).

LDAP Directory Entry Example
dn: cn=Mahansh Gaur,ou=students,dc=rtu,dc=ac,dc=in
cn: Mahansh Gaur
mail: mahansh@rtu.ac.in
uid: 2021cs001
userPassword: {SHA}...
Used by: Windows Active Directory, Linux LDAP auth, corporate email systems

DCE CDS and CORBA Naming Service

Both DCE CDS (Cell Directory Service) and CORBA Naming Service map logical names to object references/addresses. They provide location transparency — clients use names, not network addresses.

RPC — Remote Procedure Call

🔮
2026 Prediction — CERTAIN: RPC/RMI appeared in ALL THREE papers (2023 Q3, 2024 Q4, 2025 Q3). The most certain Part A topic in DS. Must know: definition, steps in RPC, stub mechanism, and RPC vs RMI difference.

What is RPC?

Remote Procedure Call (RPC) is a protocol that allows a program to call a procedure (function) on a remote machine as if it were a local call. The programmer writes code using normal function call syntax — the RPC framework handles all the network communication, parameter marshalling, and error handling transparently.

RPC is like using a TV remote control. You press "Volume Up" on your remote (client code). You don't need to know whether the TV uses infrared or Bluetooth signals (the network layer). The TV receives the signal and executes the volume change (server procedure). From your perspective, you just pressed a button and the TV responded — exactly like pressing a button on the TV itself. RPC makes remote function calls feel exactly like local function calls.

How RPC Works — The Stub Architecture

Fig: RPC call flow — programmer only writes steps 1 and 7; everything else is handled by the RPC framework

RPC Steps in Detail

8 Steps of RPC Execution
1. CLIENT CALL: Client code calls procedure normally: result = add(3, 4)
2. CLIENT STUB PACKS (Marshalling): Stub packs params into a message
• Converts data to machine-independent format (XDR — External Data Representation)
• Handles endianness, type conversion between different architectures
3. CLIENT OS SENDS: Stub sends message to server OS (via socket/UDP)
4. CLIENT BLOCKS: Client process waits for reply
5. SERVER OS RECEIVES: Server OS receives the message
6. SERVER STUB UNPACKS (Unmarshalling): Extracts parameters from message
7. ACTUAL EXECUTION: Server calls the real procedure with unpacked params
8. REPLY SENT BACK: Server stub marshals result → sends back → client stub unpacks → client continues
Binding: Client must first locate the server. Uses a Binder/Name Service (port mapper in Sun RPC) to find server's address.

RPC Parameter Passing — Marshalling

Marshalling (Parameter Serialization)
Problem: Different machines have different:
• Integer sizes (32-bit vs 64-bit)
• Byte order (big-endian SPARC vs little-endian x86)
• Floating point formats (IEEE 754 vs others)
• String encodings (ASCII vs UTF-8 vs UTF-16)
Solution: XDR (eXternal Data Representation) — canonical format
Sender converts to XDR; receiver converts from XDR
Modern: Protocol Buffers (protobuf), MessagePack, JSON, XML
Marshalling = packing. Unmarshalling = unpacking. Also called serialization/deserialization.

RPC Failure Semantics

What happens when the network or server fails during an RPC call? The client may not know if the procedure executed or not. Four possible semantics:

SemanticGuaranteeHow AchievedUse Case
MaybeRPC may or may not executeNo retry, no acknowledgementNon-critical operations
At Least OnceRPC executes one or more timesKeep retrying until reply receivedIdempotent operations (read)
At Most OnceRPC executes 0 or 1 timesServer keeps history; detects duplicatesNon-idempotent (financial transactions)
Exactly OnceRPC executes exactly onceTransactions + 2PC; very expensiveBanking, booking systems

Q2(Part B). Explain transport level communication services for building distributed applications.
2023 Part B Q2 · 4 marks ⭐⭐Unit 3Transport Services
Transport Layer Services for Distributed Applications:

1. TCP (Transmission Control Protocol): Connection-oriented, reliable, ordered, error-checked byte stream. 3-way handshake (SYN → SYN-ACK → ACK). Guarantees delivery and ordering. Used by: HTTP/HTTPS, FTP, RPC (traditional), RMI. Overhead: connection setup, ACKs, retransmission.

2. UDP (User Datagram Protocol): Connectionless, unreliable, unordered datagrams. No handshake. Fast — no overhead. Used by: DNS, streaming (video/audio), gaming, NFS (historically). Application must handle losses/ordering.

Services provided by transport layer:
· Addressing (port numbers — identify service on a host)
· Multiplexing (multiple applications share one IP address)
· Error detection (checksum)
· Flow control (TCP: don't overwhelm receiver)
· Congestion control (TCP: don't overwhelm network)

For distributed applications: RPC typically uses TCP for reliability. Real-time services use UDP for speed and tolerate losses.

RMI — Remote Method Invocation

What is RMI?

Remote Method Invocation (RMI) is the Java-specific mechanism for calling methods on objects that reside on remote JVMs (Java Virtual Machines). It extends the local Java object model to a distributed setting — you can call a method on a remote Java object exactly as you would on a local object.

If RPC is like calling a function in another building via intercom, RMI is like having a complete remote-controlled robot (Java object) in the other building. You control the robot (remote object) from your side — you can call any of its methods (shoot, walk, pick up) and get back full objects as results (not just simple numbers). The robot uses Java's own internal language (serialization) to communicate with you.

RMI Steps — How to Set Up Java RMI

Java RMI Implementation Steps
1. Define Interface (extends java.rmi.Remote):
interface Calculator extends Remote {
int add(int a, int b) throws RemoteException;
}
2. Implement on Server (extends UnicastRemoteObject):
class CalculatorImpl extends UnicastRemoteObject implements Calculator {
public int add(int a, int b) { return a + b; }
}
3. Register with RMI Registry:
Naming.rebind("Calculator", new CalculatorImpl());
4. Client Lookup and Call:
Calculator c = (Calculator) Naming.lookup("rmi://server/Calculator");
int result = c.add(3, 4); // Looks local, executes remotely!
Java serialization handles object marshalling automatically — pass any Serializable object as parameter or return value.

Sockets & Transport Communication

What is a Socket?

A socket is a software endpoint for sending and receiving data over a network. Every socket is identified by an (IP address, port number) pair. Sockets are the foundation of all network programming — TCP, UDP, RPC, and HTTP are all built on sockets.

A socket is like a telephone line. The IP address is like a country+area code (identifies the building/machine), and the port number is like an extension number (identifies the specific department/service in that building). When you call port 80 on an IP address, you reach the web server. Port 25 reaches the email server. Every service has its own "extension number."

TCP vs UDP Sockets

Fig: TCP Socket — connection-oriented; server binds and listens; client connects; bidirectional data exchange; both close when done

Connection-Less Communication Using Sockets — UDP (2023 Part B Q5)

UDP Socket — Connectionless
No connection establishment (no 3-way handshake)
Client: socket() → sendto(data, server_addr) → recvfrom()
Server: socket() → bind(IP, port) → recvfrom() → sendto(reply, client_addr)
Characteristics:
• Each datagram independently addressed and routed
• No guarantee of delivery, order, or non-duplication
• Low overhead — no connection state maintained
• Supports multicast and broadcast (TCP cannot)
• Application must implement reliability if needed
Use cases: DNS (single query/response), video streaming, SNMP, NFS, real-time gaming, VoIP

Connectionless Communication using UDP Sockets:

In connectionless communication, each message (datagram) is independently sent to the destination without establishing a persistent connection first.

Server Steps:
1. socket(AF_INET, SOCK_DGRAM) — create UDP socket
2. bind(IP, port) — bind to well-known port
3. recvfrom(&data, &client_addr) — receive datagram (contains sender address)
4. Process request
5. sendto(&reply, client_addr) — send reply to client's address

Client Steps:
1. socket(AF_INET, SOCK_DGRAM) — create UDP socket
2. sendto(&request, server_addr) — send datagram directly (no connect needed)
3. recvfrom(&reply) — receive reply

Properties: No connection overhead, lightweight, supports multicast. Application must handle lost/reordered packets. Example: DNS uses UDP — one query, one reply; no persistent connection needed.

Explain Inter-Process Communication Mechanism in detail.

VERY HIGH — 2 consecutive years

IPC (Inter-Process Communication) enables processes to communicate and synchronize.

1. Pipes: Unidirectional byte stream. Anonymous (parent-child) or Named (FIFO)
2. Message Queues: Buffered async messages. Decoupled. Ex: Kafka, RabbitMQ
3. Shared Memory: Fastest IPC — map same physical memory. Needs semaphore.
4. Sockets: TCP (reliable) or UDP (fast). Cross-machine. Foundation of DS comms.
5. RPC/RMI: High-level — hides socket complexity. Looks like local function call.
6. Signals: Async process notification (SIGKILL, SIGINT). Local only.
In DS, sockets → message passing → RPC dominate. Shared memory = local only.

📖 Part C — 10 marks (Q2)
Unit 3 | Synchronization
Explain the language mechanisms for synchronization in concurrent distributed systems.
Language mechanisms appeared in 2025 Part C Q2. A strong Part C question. Based on RTU's confirmed pattern (topics in 2025 Part C return in 2026), HIGH confidence for 2026 Part C. Draw semaphore + monitor diagrams.
HIGH — 2025 Part C anchor

Need: Concurrent processes accessing shared resources → race conditions. Language mechanisms prevent incorrect concurrent execution.

1. Semaphores (Dijkstra): Integer variable; wait(S)/signal(S) atomic operations. Binary (mutex); Counting (N resources). P(S): S--; if S<0 block. V(S): S++; wake waiting process.

2. Monitors: High-level OO construct. Shared data + procedures encapsulated. Only ONE active at a time (automatic mutual exclusion). condition.wait() releases monitor + sleeps; condition.signal() wakes one waiter. Java synchronized keyword = monitor.

3. Message Passing (for DS without shared memory): Synchronization through send()/receive(). Receiver synchronizes with sender implicitly. ADA Rendezvous: client calls entry; server accepts; both block until rendezvous; execute together.

4. Mutex + Condition Variables (POSIX): lock(mutex) before CS; unlock(mutex) after. cond_wait() = release lock + sleep atomically; cond_signal() = wake one thread.

Semaphore: low-level, manual pair — error-prone
Monitor: high-level, automatic ME — safer, preferred
Message Passing: for DS (no shared memory) — primary mechanism
