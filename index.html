📝 Part A — 2 marks (Q1)
Unit 2 | Introduction
What is a Distributed System? (🔮 PREDICTED)
Asked at Q1 in 2024 and Q1 in 2025 — two consecutive years. This is the strongest Part A anchor in DS. All 2026 RTU papers confirmed that Q1 is always the subject definition. CERTAIN.
CERTAIN

A Distributed System is a collection of independent computers (nodes) connected by a network that appears to users as a single coherent system. Each node has its own processor and memory — there is no shared clock or shared memory across nodes.

Formal Definition (Tanenbaum)
"A distributed system is one in which the failure of a computer
you didn't even know existed can render your own computer unusable."
Key: Independent machines → appear as one → transparency to user

Layman: When you use Google Docs, your document is stored on hundreds of servers worldwide. You see one document, but behind the scenes thousands of computers work together. That's a distributed system — many machines, one experience.

Examples: World Wide Web, Google Search, ATM networks, airline reservation systems, WhatsApp messaging infrastructure.

📝 Part A — 2 marks (Q2)
Unit 2 | Introduction
What are the key features of a distributed system? (🔮 PREDICTED)
Key features asked in 2025 Q2. Features/characteristics/goals alternate at Q2 slot — 2023 had goals, 2025 had features. 2026 will test one of these — features is most likely given the 2025 phrasing.
VERY HIGH confidence

Key features of a Distributed System:

  • Transparency: Hides the distributed nature from users — they see one system (location, replication, failure transparency)
  • Scalability: Can grow by adding more nodes without disrupting existing service
  • Fault Tolerance: System continues working even when some components fail
  • Concurrency: Multiple processes execute simultaneously across nodes
  • No shared clock: Each node has its own clock — no global time (Lamport's clocks solve this)
  • No shared memory: Nodes communicate only by message passing over the network
  • Heterogeneity: Nodes can be different hardware/OS — connected by middleware
  • Openness: Can be extended and accessed using standard protocols and interfaces

RPC vs RMI — Key Differences
Feature | RPC | RMI
────────────────────────────────────────────────
Full form | Remote Procedure Call | Remote Method Invocation
Language | Language-independent (C) | Java-specific
Paradigm | Procedural programming | Object-oriented programming
What is called | Remote procedure/function| Method on remote object
Object passing | Cannot pass objects | Can pass objects by value/ref
Protocol | Uses stubs/skeletons | Uses stubs/skeletons + ORB
Example | Sun RPC, DCE RPC | Java RMI, CORBA

Layman: RPC = calling a function on another machine (like pressing a remote button that triggers something far away). RMI = calling a method on an object that lives on another machine — more powerful because you work with complete objects, not just functions.

📝 Part A — 2 marks (Q4)
Unit 4 | Process Scheduling
Define Load Balancing in a distributed system. (🔮 PREDICTED)
Load balancing appeared in 2025 Q4 as a new Part A question. RTU 2026 patterns (CAO, ISS, CC) all confirmed that new 2025 Part A topics are retained. Load balancing is also a core DS concept — very likely to stay.
VERY HIGH confidence

Load Balancing in a distributed system is the process of distributing computational workload evenly across multiple nodes so that no single node is overloaded while others are idle. It maximises resource utilisation and minimises response time.

Goal: Distribute work so all nodes have approximately equal load
Metric: CPU utilisation, queue length, response time
Types:
Static: Workload assigned at compile/start time (no runtime adjustment)
Dynamic: Workload redistributed at runtime based on current load (more flexible)
Layman: Like a restaurant manager who monitors all tables — when one waiter is overwhelmed, they reassign customers to an idle waiter. Result: all waiters equally busy, customers served faster.

Algorithms: Round Robin, Least Loaded First, Sender-Initiated (busy node pushes work out), Receiver-Initiated (idle node pulls work in).

📝 Part A — 2 marks (Q6)
Unit 4 | Process Scheduling
List two advantages of dynamic load sharing. (🔮 PREDICTED)
Dynamic load sharing appeared in 2025 Q6. RTU tends to follow new topics with follow-up questions. 2026 may continue with this topic in Part A — advantages are a natural continuation from defining it.
HIGH confidence

Dynamic Load Sharing allows processes to be moved between nodes at runtime based on current system load. Advantages:

  • Adaptive response: Responds to changing workloads in real-time — if a node becomes overloaded, tasks are immediately moved to lighter nodes. No pre-planning needed.
  • Better resource utilisation: Prevents nodes from being idle while others are overloaded — maximises use of available CPU across the entire cluster.
  • Improved throughput: Reduces job completion time by avoiding bottlenecks — jobs finish faster when load is spread evenly.
  • No prior knowledge needed: Works without knowing task execution times in advance — unlike static scheduling which needs this information.
📝 Part A — 2 marks (Q7)
Unit 5 | DSM
What is Distributed Shared Memory (DSM)? (🔮 PREDICTED)
DSM appeared in 2023 Q7 (NUMA), 2025 Q7, and 2025 Q8. Two of three papers have DSM in Q7/Q8 slot. The 2026 DIP/CC patterns confirm: Unit 5 DSM topics are tested every year in Part A. CERTAIN for 2026.
CERTAIN

Distributed Shared Memory (DSM) is an abstraction that gives processes on different nodes the illusion of a single shared memory address space, even though physically each node has its own local memory and they communicate over a network.

Physical reality: Node A has RAM A, Node B has RAM B — separate memory
DSM illusion: Both nodes see one big shared virtual address space
Access: If Node A reads address X that lives on Node B → network fetch
Layman: DSM is like pretending all the books in different libraries are on ONE big shared shelf. When you want a book from another library, the librarian secretly fetches it — you don't need to go there yourself.

Advantage: Simplifies programming — programmers don't need to write explicit message passing code. Disadvantage: Network accesses for non-local data are much slower than local memory access.

📝 Part A — 2 marks (Q8)
Unit 5 | DSM
How does Distributed Shared Memory improve system performance? (🔮 PREDICTED)
Appeared in 2025 Q8 — directly paired with Q7's DSM definition. RTU frequently pairs definition (Q7) with advantages/how-it-works (Q8). Memory consistency models appeared at Q5 in 2025 and are also strong candidates for Q8 in 2026.
HIGH confidence

DSM improves performance through several mechanisms:

  • Caching: Frequently accessed shared data is cached locally at each node — reduces expensive network fetches. A page accessed repeatedly stays in the local cache.
  • Replication: Read-only pages can be replicated across multiple nodes — each node serves read requests locally without network communication.
  • Coherence protocol optimisation: Smart invalidation protocols (like MESI) minimise coherence traffic — only notify nodes that actually have a copy when a write occurs.
  • False sharing reduction: Grouping related data on the same page reduces the need to transfer pages between nodes.
  • Simplified programming model: Programmers write shared-memory code — the DSM system handles communication automatically, often more efficiently than hand-coded message passing.

Unit 5 + 6 | Computation + Agreement
What is the concept of Faults in a distributed system? (🔮 PREDICTED)
Faults appeared in 2023 Q9 (faults in distributed agreement) and 2025 Q9. Present in 2 of 3 papers. Faults/failures directly lead to Byzantine Agreement (the most tested Part C topic) — RTU pairs these topics consistently.
VERY HIGH confidence

In distributed systems, faults, errors, and failures form a causal chain:

Fault → Error → Failure Chain
Fault: Root cause — the defect (hardware bug, software error, cosmic ray)
Error: Manifestation — incorrect system state caused by the fault
Failure: Observable effect — system deviates from specified behaviour
Example: Bad memory chip (fault) → corrupted data (error) → wrong output (failure)

Types of Faults in DS:

  • Transient: Occurs once and disappears — cosmic ray flipping a bit
  • Intermittent: Occurs, disappears, recurs — loose connector
  • Permanent: Persists until repaired — burned-out chip
  • Byzantine (Arbitrary): Node fails in completely unpredictable ways — may send conflicting messages to different nodes (the hardest to tolerate)
📝 Part A — 2 marks (Q10)
Unit 6 | Replicated Data
What are the key issues in managing replicated data? (🔮 PREDICTED)
Replicated data issues appeared in 2025 Q10 — new topic. RTU 2026 patterns confirmed Q10 anchors persist year to year (CC had "layers + service models" at Q10 both 2025 and 2026 predictions). Replicated data is core Unit 6 — CERTAIN.
VERY HIGH confidence

Key issues in managing replicated data:

  • Consistency: All replicas must show the same value — when one is updated, others must be updated too. Strong consistency = all reads see latest write (expensive). Eventual consistency = replicas converge over time (cheaper).
  • Update propagation: How quickly should updates reach all replicas? Synchronous (immediate, slower) vs asynchronous (delayed, faster)?
  • Conflict resolution: When two nodes update the same data simultaneously, whose update wins? (Last-writer-wins, merge, application-specific logic)
  • Replica placement: Where to place replicas for optimal access time and fault tolerance?
  • Failure handling: If a replica becomes unavailable, how do the remaining replicas continue to serve requests?
  • Read vs Write trade-off: More replicas = faster reads but more complex writes (more replicas to update)
📋 Part B — 4 marks (Q1)
Unit 2 | OS Types
Discuss different types of operating systems used in a distributed system. (🔮 PREDICTED)
Types of operating systems appeared in 2025 Part B Q1. RTU retained new 2025 Part B Q1 topics in subsequent years (CAO, CC confirmed). OS types (centralized, network, distributed, cooperative) are directly from the syllabus Unit 2.
HIGH confidence

Four types of operating systems related to distributed systems:

  • Centralised OS: One powerful computer with all resources centralised. Users access via terminals. Examples: Mainframe OS. Simple to manage, but single point of failure. No distribution — all in one place.
  • Network OS (NOS): Each machine has its own OS. Resources shared over network but NOT transparent — users must explicitly know where files/printers are. Examples: Windows Server, Novell NetWare. Not truly distributed — just networked.
  • Distributed OS: Single OS image across multiple machines — users see one unified system. Resources are transparently shared. Examples: Amoeba, Plan 9. Tight coupling — looks like one machine.
  • Cooperative Autonomous Systems: Independent systems that cooperate to solve problems. Each has full autonomy but coordinates with others. Examples: Grid computing, SETI@home. Loosely coupled — each machine is fully independent.
📋 Part B — 4 marks (Q2)
Unit 2 | DCE
Explain the role of Distributed Computing Environment (DCE). (🔮 PREDICTED)
DCE appeared in 2024 Q2 as "define DCE" and 2025 Part B Q2 as "explain role". Two consecutive years — confirmed anchor. 2026 DIP/CC patterns show two-consecutive-year Part B topics are certain for 2026.
CERTAIN

DCE (Distributed Computing Environment) is a middleware framework developed by the Open Software Foundation (OSF) that provides services and tools for building distributed applications across heterogeneous networks. DCE's role is to be the "glue" that connects different computers and operating systems so applications can work transparently.

Key Components and their roles:

  • Threads Service: Portable thread management across different OS platforms — enables concurrent programming in distributed apps
  • RPC (Remote Procedure Call): Core mechanism for distributed communication — allows calling procedures on remote machines transparently
  • Directory Service (Cell Directory Service — CDS): Naming service — clients locate remote services by name without knowing their network address
  • Security Service: Authentication + authorisation + encryption — Kerberos-based security across the distributed environment
  • Distributed File Service (DFS): Unified file system view across all machines — location transparency
  • Time Service (DTS): Synchronises clocks across distributed nodes — essential for event ordering and consistency
DCE Architecture (layers, bottom to top):
Hardware → OS → DCE Middleware → Distributed Applications
DCE hides heterogeneity — apps talk to DCE, not directly to different OS types
📋 Part B — 4 marks (Q3)
Unit 3 | IPC
Explain Inter-Process Communication (IPC) mechanism in detail. (🔮 PREDICTED)
IPC appeared in 2025 Part B Q3. IPC is fundamental to ALL distributed system communication — it cannot be absent from a DS exam. The 2025 phrasing "explain IPC mechanism" is predicted to return or appear in a closely related form.
VERY HIGH confidence

IPC (Inter-Process Communication) refers to mechanisms that allow processes on the same or different machines to communicate, coordinate, and share data.

IPC Mechanisms in Distributed Systems:

  • Message Passing: Processes exchange discrete messages — send() and receive() primitives. Can be synchronous (blocking — sender waits for receiver) or asynchronous (non-blocking — sender continues immediately). Foundation of all distributed communication.
  • Remote Procedure Call (RPC): Higher-level abstraction over message passing — makes remote communication look like local procedure calls. Client stub marshals parameters → sends to server → server stub unmarshals → executes → returns result.
  • Sockets: Low-level communication endpoint — identified by (IP address, port number). TCP sockets = connection-oriented (reliable). UDP sockets = connectionless (fast, unreliable).
  • Shared Memory (in DSM): Virtual shared address space — processes communicate by reading/writing shared memory locations. Fast but requires synchronisation.
  • Request/Reply: Client sends request message → server processes → returns reply. Simple, stateless interaction — basis of HTTP.
📋 Part B — 4 marks (Q4)
Unit 5 | Computation Models
What is distributed deadlock handling? Explain the three strategies. (🔮 PREDICTED)
Deadlock handling appeared in 2023 Part C Q3 (all three strategies) and 2025 Part B Q4. Moving from Part C to Part B format is a common RTU pattern. 2026 likely tests deadlock in Part B with the three strategies (prevention, avoidance, detection + recovery).
VERY HIGH confidence

Distributed Deadlock occurs when two or more processes in a distributed system are each waiting for resources held by the other, creating a circular wait with no progress possible.

Three Strategies for Deadlock Handling:

  • 1. Deadlock Prevention: Eliminate one of the four Coffman conditions (mutual exclusion, hold and wait, no preemption, circular wait). Example: Require processes to request ALL resources at once (eliminates hold-and-wait) — simple but wasteful.
  • 2. Deadlock Avoidance (Banker's Algorithm): Before granting a resource, check if the system remains in a "safe state" — a state from which all processes can eventually complete. If granting the resource leads to an unsafe state, deny the request. Requires knowing maximum resource needs in advance.
  • 3. Deadlock Detection and Recovery: Allow deadlocks to occur, detect them using Wait-For Graphs (WFG), then break the deadlock by aborting one or more processes. In distributed systems, each node maintains a local WFG; a coordinator combines all local WFGs to detect global cycles.
Wait-For Graph (WFG): Directed graph where P→Q means "P is waiting for Q to release resource"
Deadlock = cycle in WFG
Example: P1→P2→P3→P1 = deadlock (circular wait)
📋 Part B — 4 marks (Q5)
Unit 4 | DFS
Explain the distributed process implementation. (🔮 PREDICTED)
Distributed process implementation appeared in 2025 Part B Q5. 2026 DIP/CAO patterns confirm new Part B topics from 2025 persist. Process implementation covers how processes are created, managed, and migrated across nodes.
HIGH confidence

Distributed Process Implementation refers to how processes are created, placed, and managed across multiple nodes in a distributed system.

Key aspects:

  • Process Creation: When a new process is needed, the system decides which node should run it (static assignment) or the process is created locally and may migrate later (dynamic assignment).
  • Process Migration: Moving a running process from one node to another — reasons: load balancing, better resource access, fault tolerance. Requires transferring the full process state (registers, memory, open files, network connections).
  • Static Process Scheduling: Process-to-node assignment decided before execution begins, based on communication patterns. Tasks that communicate frequently are placed on the same or nearby nodes — reduces network overhead.
  • Dynamic Load Sharing: Processes may be sent from heavily loaded nodes to lightly loaded nodes at runtime. Two approaches: sender-initiated (overloaded node pushes work out) and receiver-initiated (idle node pulls work in).
  • Process Representation: Processes described by their code + data + execution state. Process Graph Models show dependencies between processes — used for scheduling decisions.
📋 Part B — 4 marks (Q6)
Unit 5 | Computation Models
Explain the concept of memory consistency models. Why are they important? (🔮 PREDICTED)
Memory consistency appeared in 2025 Part B Q6. This directly relates to DSM — you cannot understand DSM without understanding consistency models. RTU tests both DSM and consistency in tandem. Very likely for 2026.
HIGH confidence

A memory consistency model defines the order in which memory operations (reads and writes) appear to execute to different processors in a shared memory or DSM system. It answers the question: "If process A writes to memory location X, when will process B's read of X see the updated value?"

Why they are important: Without a defined model, programmers cannot reason about the correctness of their concurrent programs. Different models offer different trade-offs between consistency strength and performance.

Key Consistency Models:

  • Strict Consistency: All writes are immediately visible to all processes. Like a single-processor — ideal but impossible in distributed systems (requires global clock).
  • Sequential Consistency (Lamport): The result of execution is the same as if all operations were executed in some sequential order, and each process's operations appear in program order. Achievable but expensive.
  • Causal Consistency: Writes that are causally related must be seen in the same order by all processes. Concurrent writes may be seen in different orders. Better performance than sequential.
  • Eventual Consistency: If no new updates occur, all replicas eventually converge to the same value. Used by Amazon DynamoDB, DNS. High availability, low consistency.
📋 Part B — 4 marks (Q7)
Unit 6 | Agreement
Discuss the Randomized Distributed Agreement approach and its advantages. (🔮 PREDICTED)
Randomized distributed agreement appeared in 2025 Part B Q7. This is a direct syllabus topic (Unit 6). RTU tends to repeat exact Part B Q7 topics — 2026 DIP, CC patterns confirmed B-Q7 persistence. High confidence.
HIGH confidence

The Randomized Distributed Agreement approach uses probabilistic algorithms to achieve consensus in distributed systems, especially in the presence of Byzantine faults where deterministic consensus is impossible (FLP impossibility theorem).

Why randomisation? The FLP (Fischer-Lynch-Paterson) theorem proves that deterministic consensus is impossible in an asynchronous distributed system with even one crash failure. Randomisation circumvents this impossibility by allowing probabilistic termination — the algorithm may not terminate with probability 1 in finite time, but terminates with high probability.

Randomized Agreement Algorithm (Ben-Or Protocol):

  • Each process proposes a value (0 or 1)
  • Processes broadcast their proposed values and collect votes
  • If a majority agrees on one value → adopt it. If split → randomly choose a value and retry
  • Eventually all processes converge on the same value with probability 1

Advantages of Randomized Approach:

  • Circumvents FLP impossibility — achieves consensus that deterministic algorithms cannot
  • Tolerates Byzantine failures without requiring synchrony assumptions
  • Simple to implement compared to deterministic Byzantine agreement algorithms
  • Expected termination in O(1) rounds for benign failures
📖 Part C — 10 marks (Q1)
Unit 2 | Architecture
Explain the Architecture Models of a Distributed System. How do design issues affect system performance? (🔮 PREDICTED)
Architecture Models appeared in Part C Q1 in 2025. "Need of DS + characteristics" appeared in Part C Q1 in 2024. "Client-server + RPC" appeared in Part C Q2 in 2024. Architecture is the most tested Part C topic. CERTAIN for 2026 Q1.
CERTAIN — Every Paper in some form

Architecture Models define how components of a distributed system are structured, interact, and are organised across machines. The architecture directly determines performance, scalability, and fault tolerance.

Major Architecture Models:

  • Client-Server Model: Most common. Server provides services; clients consume them. Client sends request → server processes → returns response. Centralised service management. Examples: Web servers, database servers, email servers.
  • Peer-to-Peer (P2P): All nodes are equal — each can be both client and server. Highly scalable, no single point of failure. Examples: BitTorrent, Bitcoin, Gnutella.
  • Three-Tier Architecture: Presentation layer (client) + Application layer (middle tier/server) + Data layer (database). Separates concerns — scales each tier independently. Examples: Web applications.
  • Multi-tier (N-tier): Extension of three-tier with more separation. Enterprise applications often use 4+ tiers.
  • Mobile Code Model: Code moves from server to client and executes there. Example: Java applets, JavaScript. Reduces server load, but raises security concerns.
Fig — Left: Client-Server (centralised, single point of failure) vs Right: P2P (decentralised, highly fault-tolerant)

Design Issues and their impact on performance:

  • Communication latency: Network delays add overhead — design must minimise network round trips (batching requests, caching)
  • Naming and location transparency: Finding resources efficiently impacts throughput — poor naming = slow lookups
  • Fault tolerance: Replication improves availability but adds consistency overhead — trade-off between availability and performance
  • Scalability: Centralised components become bottlenecks — distributed design avoids hot-spots
  • Synchronisation: Coordinating processes adds overhead — minimise synchronisation points
📖 Part C — 10 marks (Q2)
Unit 3 | Synchronization
Explain the language mechanisms for synchronization in distributed systems. (🔮 PREDICTED)
Language mechanisms for synchronization appeared in 2025 Part C Q2 — a new Part C topic. Based on 2026 CC/ISS pattern of retaining new Part C topics, this is predicted for 2026. Covers semaphores, monitors, message passing for synchronization.
HIGH confidence

Language mechanisms for synchronization are programming language constructs and runtime primitives that enable concurrent processes to coordinate access to shared resources and communicate safely.

1. Semaphores: Integer variable with two atomic operations — wait(S) and signal(S). Oldest synchronisation mechanism.

wait(S): if S > 0 then S-- else block the process
signal(S): if waiting processes exist then wake one else S++
Problem: Semaphores are low-level — easy to misuse, leads to deadlocks if wait/signal are in wrong order

2. Monitors (Hoare): High-level synchronisation construct — a module that encapsulates shared data with procedures that provide mutually exclusive access. Only one process can be active in a monitor at a time. Condition variables (wait and signal) allow blocking within the monitor.

3. Conditional Critical Regions: A region of code that can only execute when a specific condition is true AND the shared variable is not in use. Compiler ensures mutual exclusion automatically.

4. Message Passing: In distributed systems where shared memory doesn't exist, synchronisation is achieved through message passing — send() and receive() serve as synchronisation points. Rendezvous synchronisation: sender and receiver both block until the other is ready.

5. Mutex (Mutual Exclusion Object): Binary semaphore with ownership — only the process that locked it can unlock it. Used to protect critical sections in multi-threaded distributed servers.

6. Barrier Synchronisation: All participating processes must reach the barrier point before any can proceed — used in parallel computations to ensure all partial results are ready before combining them.

Unit 4 | DFS
Discuss the design and implementation of a Distributed File System. (🔮 PREDICTED)
DFS appeared in 2023 Part A Q4, 2024 Part A Q5, and 2025 Part C Q3. Present in every paper across different parts. Part C Q3 is its natural slot for 2026 — covers the full DFS design (naming, caching, replication, consistency, failure handling).
VERY HIGH confidence

A Distributed File System (DFS) provides transparent file storage and access across multiple nodes — users and applications access files as if they were local, regardless of their actual physical location.

Transparencies that DFS must provide:

  • Location transparency: Users don't know which server stores the file — access by name, not location
  • Access transparency: Local and remote files accessed the same way — same API
  • Replication transparency: Multiple copies may exist — users see one file
  • Failure transparency: If one server fails, another serves the file — users unaffected

DFS Design Components:

  • Naming and Location Service: Maps file names to physical locations. Flat name space vs hierarchical. Example: /home/user/file.txt → located on Server 3, block 42.
  • Client Caching: Clients cache recently accessed files locally — reduces network traffic. Cache consistency challenge: local cache must be validated before use (write-through vs write-back).
  • File Replication: Multiple copies stored on different servers — improves availability and read performance. Primary-backup replication or quorum-based replication.
  • Concurrency Control: Multiple clients may access the same file — locking or optimistic concurrency (read-modify-write with version check).
  • Stateful vs Stateless servers: Stateless (like NFS v2) = server doesn't remember client state — simpler crash recovery. Stateful = server remembers open files — better performance.

Case Study — Andrew File System (AFS): Developed at CMU. Whole-file caching on client. Files cached locally for the entire session — dramatically reduces server load. Callback mechanism: server notifies clients when a cached file is modified by another client.

Case Study — Coda File System: Extension of AFS. Supports disconnected operation — laptop users can work offline using cached copies. When reconnected, changes are propagated back (reintegration). Uses volume storage groups for replication.

Unit 5 | DSM
Explain the implementation of Distributed Shared Memory (DSM) systems. (🔮 PREDICTED)
DSM implementation appeared in 2025 Part C Q4. Based on CC and ISS 2026 patterns, new 2025 Part C topics are almost always retained in 2026. DSM implementation (page-based, object-based, coherence protocols) is core Unit 5 material.
VERY HIGH confidence

DSM Implementation creates the illusion of shared memory across multiple nodes that physically have separate memories. The implementation must handle: where data is located, how it's accessed, and how consistency is maintained.

Implementation Approaches:

  • Page-Based DSM: Memory divided into pages (like virtual memory). When a process accesses a page not in local memory, a page fault occurs — the DSM runtime fetches the page from the remote node that owns it. Example: IVY system. Simple (reuses VM hardware) but causes false sharing — two variables on same page force unnecessary page transfers.
  • Object-Based DSM: Shared objects (not pages) are the unit of sharing. Objects migrate when accessed remotely. Finer granularity than page-based — less false sharing. Example: Emerald language.
  • Variable-Based DSM: Individual shared variables tracked — finest granularity but highest overhead.

Coherence Protocols (managing consistency):

  • MESI Protocol (for pages/cache lines):
    • Modified (M): Page locally modified, only copy
    • Exclusive (E): Unmodified, only copy
    • Shared (S): Unmodified, multiple copies exist
    • Invalid (I): Page is stale — must fetch fresh copy
  • Invalidation protocol: When a write occurs, all other copies are invalidated. Next reader must fetch fresh copy. Simple but generates many messages on write-sharing.
  • Update protocol: When a write occurs, all copies are updated simultaneously. All copies stay fresh. Better for frequently-shared read-mostly data.
Page-Based DSM Access Flow:
Process on Node A accesses address X → not in local memory → PAGE FAULT
DSM runtime: "Which node owns page containing X?" → ask directory
Directory: "Node B owns it" → send request to Node B
Node B: transfers page to Node A, marks its copy Invalid
Node A: maps page into local memory → process resumes

Unit 6 | Agreement
What is Byzantine Agreement? Explain different approaches to solve it. (🔮 PREDICTED)
Byzantine Agreement appeared in Part C in 2024 (Q5) and 2025 (Q5) — TWO CONSECUTIVE YEARS in the SAME SLOT. This is the most certain Part C prediction in DS — identical to A* in AI and Booth's in CAO. Always explain with the "Generals" analogy + the f < n/3 condition.
CERTAIN — 2 consecutive years Part C Q5

Byzantine Agreement (Byzantine Generals Problem) was formulated by Lamport, Shostak, and Pease (1982). It asks: how can distributed processes reach consensus (agree on the same value) when some processes may be faulty and behave arbitrarily — including sending different values to different processes (Byzantine fault)?

The Generals Analogy: Imagine N generals surrounding a city, communicating only by messengers. They must agree unanimously whether to attack or retreat. Some generals may be traitors who send conflicting orders. The problem: achieve consensus among loyal generals despite traitors.

Byzantine Agreement Conditions
Validity: If all correct processes propose v, then all correct processes decide v
Agreement: All correct processes decide the same value
Termination: All correct processes eventually decide
Key result: Need at least n ≥ 3f + 1 processes to tolerate f Byzantine faults
Meaning: If 1 in 4 nodes is faulty (f=1, n=4) → can tolerate Byzantine failures. With only 3 nodes and 1 faulty → impossible!

Approach 1 — Oral Messages (OM) Algorithm:

  • Commander sends its value v to all lieutenants
  • Each lieutenant forwards received value to all others
  • Each lieutenant uses majority voting on received values
  • Requires n ≥ 3f + 1 processes and f+1 rounds
  • Works if message content can be changed (no authentication)

Approach 2 — Written Messages (Signed Messages) Algorithm:

  • Messages are digitally signed — recipients can verify authenticity and detect alterations
  • Requires only n ≥ f + 2 processes (much better than 3f+1)
  • Requires Public Key Infrastructure (PKI) for signatures

Approach 3 — Practical Byzantine Fault Tolerance (PBFT):

  • Efficient algorithm for distributed systems with n ≥ 3f+1 replicas
  • Uses three-phase protocol: Pre-prepare → Prepare → Commit
  • O(n²) message complexity — used in blockchain systems

Approach 4 — Randomized Byzantine Agreement:

  • Uses random coin flips to break symmetry and achieve probabilistic agreement
  • Circumvents FLP impossibility theorem — terminates with probability 1
  • Ben-Or protocol: If all agree → commit; else → flip a coin and retry

Real-world application: Blockchain (Bitcoin uses proof-of-work, Ethereum uses PBFT-variants) — Byzantine Agreement ensures all nodes agree on the same transaction history even with malicious nodes attempting to double-spend.
