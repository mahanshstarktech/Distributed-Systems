Concept of Faults, Failure & Recovery

🔮

The Three-Level Model: Fault → Error → Failure

These three terms are often confused but have precise meanings in distributed systems. Understanding this hierarchy is essential for both Part A and Part B answers.

Think of a car. A crack in the brake pad (FAULT — root cause, hidden). The brake fluid starts leaking because of the crack (ERROR — the bad system state resulting from the fault). The car cannot stop when you press the brake (FAILURE — the visible, externally observable bad behaviour). The fault was there long before the failure was noticed.
The Chain: Fault → Error → Failure
FAULT = The underlying defect (root cause). May be dormant for a long time.
Example: A bug in code, a corrupted memory bit, a loose network cable
ERROR = An incorrect state that results when a fault is activated.
Example: A variable holds the wrong value due to the bug being triggered
FAILURE = When the system deviates from its specified behaviour — visible to users.
Example: The server returns wrong data; the process crashes; request times out
Memory: "FEF" — Fault causes Error causes Failure. The fault is private (inside), failure is public (visible outside).

Types of Failures (Process/Component Level)

A process can fail in different ways — listed from easiest to handle to hardest to handle:

5 Failure Types (Crash → Byzantine) — Increasing Severity
1. CRASH-STOP (Fail-Stop): Process crashes and stops permanently. No more messages.
Easiest: Others can detect silence → assume crashed. No wrong answers, just silence.
Example: A server process killed by kill -9.
2. CRASH-RECOVERY: Process crashes but may restart from saved state (checkpoint).
Moderate: May have stale data on restart. Logging helps recovery.
Example: A database that restarts after a crash and replays its WAL log.
3. OMISSION FAILURE: Process omits to send/receive some messages (not all).
Still responds to some — hard to distinguish from slow or crashed process.
Example: Network packet drops; receive-omission = process ignores some requests.
4. TIMING FAILURE: Process responds, but outside the specified time bounds.
In synchronous model only (asynchronous: no timing bounds to violate).
Example: Real-time system: sensor must report in 10ms, takes 50ms.
5. BYZANTINE (Arbitrary) FAILURE: Process behaves ARBITRARILY.
Sends wrong, inconsistent, or malicious messages. HARDEST to handle.
Example: Corrupted process sends "attack" to half the generals, "retreat" to others.
Think of severity as a ladder: Crash-Stop (bottom, easiest) → Byzantine (top, hardest). Each step up is harder to detect and recover from.

Failure Recovery Mechanisms

Once a failure is detected, the system must recover. RTU syllabus covers these recovery approaches:

  • Checkpointing: Save process state periodically. On failure, restart from last checkpoint. Called backward recovery — roll back to a known good state.
  • Message Logging: Log all messages received. On recovery, replay logged messages to reconstruct state without needing sender to resend.
  • Replication: Keep multiple copies of data/processes. If one fails, another takes over. Most common approach in practice.
  • Forward Recovery: Error-correcting codes (like RAID) allow the system to reconstruct the correct state without rolling back.
Backward recovery = "undo" — go back to your last save game point. Forward recovery = "repair" — use parity bits to reconstruct the correct data without going back. Replication = "redundancy" — keep multiple copies so if one burns, another steps in.

Byzantine Faults & Adversaries

What is a Byzantine Fault?

A Byzantine fault (also called an arbitrary fault) is the most severe type of failure. A Byzantine process can:

  • Send different messages to different processes (inconsistency)
  • Send intentionally wrong messages
  • Send messages at the wrong time
  • Selectively not send messages to some processes
  • Collude with other Byzantine processes
Normal crash fault = a soldier gets injured and can't communicate at all. Everyone knows he's down (silence = crash). Byzantine fault = a TRAITOR SOLDIER who is alive and sending messages — but sending "ATTACK" to half the army and "RETREAT" to the other half. He's trying to make the army fail by creating confusion. This is why Byzantine faults are so dangerous — you can't trust anything the faulty process says.

The Byzantine Generals Problem (Original Story)

Introduced by Lamport, Shostak, and Pease (1982), this famous problem illustrates the challenge:

The Byzantine Generals Problem Setup
Scenario: Byzantine army surrounds an enemy city.
Several generals, each commanding a division, must agree on a common plan:
→ ATTACK together (they win) OR RETREAT together (they survive)
→ Some attack + some retreat = CATASTROPHIC DEFEAT
Communication: Only by messenger (can't all meet face-to-face)
Problem: Some generals may be TRAITORS (Byzantine faulty)
→ Traitors send conflicting messages to different generals
→ Goal: All LOYAL generals must agree on the SAME plan
→ Traitors try to prevent agreement or cause wrong agreement
Mapping to DS: Generals = processes, Messengers = network, Traitors = Byzantine-faulty processes. The problem is: can loyal processes agree despite traitors?

Adversary Model

An adversary is an entity (or model) representing the worst-case behaviour of faulty components. Different adversary models define different problem difficulties:

Adversary TypePowerProtocol Needed
Crash AdversaryCan crash any f processes (they stop responding)Paxos, Raft — need f+1 out of 2f+1
Byzantine AdversaryCan control f processes — they send arbitrary messagesNeed 3f+1 processes; f+1 rounds
Authenticated ByzantineByzantine but can't forge digital signaturesNeed f+2 processes (lighter requirement)
Rushing AdversaryByzantine + sees all messages before sending own (adaptive)Randomized protocols needed
⚠️
Key Insight — Why Byzantine is MUCH Harder than Crash

With crash faults, silence = crashed (binary). You know who to ignore.
With Byzantine faults, the faulty process is ACTIVE and LYING. You don't know which messages are trustworthy. That's why you need 3f+1 processes for Byzantine vs only 2f+1 for crash tolerance — the extra f processes provide the "majority margin" to outvote the liars.

Formal Definition of Byzantine Agreement

The Byzantine Agreement (BA) Problem requires that n processes, where up to f may be Byzantine faulty, reach a consensus on a single value despite the faulty processes sending arbitrary/conflicting messages.

Three Requirements of Byzantine Agreement
1. AGREEMENT: All non-faulty processes must decide on the SAME value.
(No two honest processes can decide different values)
2. VALIDITY: If all non-faulty processes propose the same value v,
then all non-faulty processes must decide v.
(Unanimous honest input must produce that result)
3. TERMINATION: Every non-faulty process eventually decides a value.
(The protocol cannot run forever)
These three properties together define a "correct" Byzantine Agreement protocol. A protocol is broken if it violates ANY one of these.

The Key Result — Minimum Processes Required

Lamport-Shostak-Pease Theorem (1982) — THE Most Important Formula
To tolerate f Byzantine faulty processes, you need at least:
n ≥ 3f + 1 total processes
Examples:
f = 1 fault: need n ≥ 4 processes (3×1+1 = 4)
f = 2 faults: need n ≥ 7 processes (3×2+1 = 7)
f = 10 faults: need n ≥ 31 processes
WHY 3f+1? Intuition:
→ f Byzantine can lie (actively mislead)
→ f may crash (silent failures)
→ Remaining n-2f non-faulty must still form a majority
→ n - 2f > f → n > 3f → n ≥ 3f+1
EXAM CRITICAL: If asked "How many processes needed?" → Answer: n ≥ 3f+1. Always verify: 4 processes with f=1: 4 ≥ 4 ✓. 3 processes with f=1: 3 < 4 ✗ IMPOSSIBLE.
Why n ≥ 3f+1? Imagine a voting system where f people are liars. If you only have 3 people (1 liar + 2 honest), the liar can tell person A "B voted YES" and tell person B "A voted YES" — now both honest people think there's a majority when there isn't. With 4 people and 1 liar: the liar can mislead one person but the remaining 2 honest people can always outvote the combined f liars + f crashers = 2f confused votes. You need n-f non-faulty (at least 2f+1) honest processes to out-majority the chaos.

Algorithm 1: Oral Messages (OM Algorithm)

The original algorithm by Lamport, Shostak & Pease. Uses recursive message passing where each general forwards what they heard to everyone else.

OM(f) Algorithm — Recursive Approach
OM(0): Commander sends value to all lieutenants. Each lieutenant uses what they received.
OM(m) where m > 0:
Step 1: Commander sends value v to each of n-1 lieutenants.
Step 2: Each lieutenant i acts as commander in OM(m-1) with the
n-2 other lieutenants, sending what they received from the commander.
Step 3: Each lieutenant i collects (n-1) values — one from original send +
(n-2) from sub-protocol. Takes MAJORITY of these values.
Example (n=4, f=1): 1 commander + 3 lieutenants (1 lieutenant is traitor)
Round 1: Commander → L1:attack, L2:attack, L3(traitor): attack
Round 2: L1 → L2:attack, L3:attack | L2 → L1:attack, L3:attack | L3 → L1:retreat, L2:retreat
L1 collects: {attack, attack, retreat} → majority = ATTACK ✓
L2 collects: {attack, attack, retreat} → majority = ATTACK ✓ (same as L1)
Agreement achieved!
Complexity: Needs f+1 rounds, n ≥ 3f+1 processes, O(n^(f+1)) messages — exponential! Practical only for very small f.

Algorithm 2: Signed Messages (SM Algorithm)

Uses digital signatures to prevent traitors from forging loyal generals' messages. This removes the need for n ≥ 3f+1 — only n ≥ f+2 needed!

Signed Messages Algorithm
Key Assumption: Digital signatures cannot be forged.
→ Loyal general's signed message cannot be altered by a traitor
→ Traitor can send different signed values to different processes,
but cannot fake a different loyal general's signature
Algorithm SM(f):
Step 1: Commander signs and sends value to all lieutenants.
Step 2: Each lieutenant maintains a set V of values it has received.
Step 3: On receiving signed message (v : signature_chain):
→ If v is new, add to V; forward to all others with your signature appended
Step 4: After f+1 rounds, decide: if |V|=1, choose that value; else choose default (RETREAT)
Minimum processes needed: n ≥ f + 2 (much better than 3f+1!)
Trade-off: Oral Messages: n ≥ 3f+1, no crypto. Signed Messages: n ≥ f+2, requires public-key infrastructure.

Algorithm 3: PBFT (Practical Byzantine Fault Tolerance)

PBFT (Castro & Liskov, 1999) is the most practical Byzantine Agreement algorithm used in real systems. It uses a 3-phase protocol.

PBFT — 3 Phase Protocol
Assumption: n ≥ 3f+1 replicas. One replica is the PRIMARY (leader); rest are BACKUPS.
Phase 1 — PRE-PREPARE:
Primary receives client request, assigns sequence number, broadcasts PRE-PREPARE msg
PRE-PREPARE(v, n, d) — view v, sequence num n, digest d (hash of request)
Phase 2 — PREPARE:
Each backup validates PRE-PREPARE; if valid, broadcasts PREPARE(v, n, d, i)
A replica is "prepared" when it has PRE-PREPARE + 2f PREPARE messages
Phase 3 — COMMIT:
Each prepared replica broadcasts COMMIT(v, n, D, i)
A replica "commits" when it has 2f+1 COMMIT messages → executes request → replies
Message complexity: O(n²) — each phase involves all-to-all communication
Used in: Hyperledger Fabric, Tendermint, early blockchain systems. Handles up to ⌊(n-1)/3⌋ Byzantine failures.
Fig: PBFT Protocol — Client request goes through Pre-Prepare → Prepare → Commit phases. Faulty replica (Replica 3) ignored after 2f+1 correct responses received.

Impossibility of Consensus (FLP Result)

The FLP Impossibility Theorem

One of the most famous results in distributed computing, proven by Fischer, Lynch, and Paterson (1985):

FLP Impossibility Result — THE Most Important Theorem
Statement: In an ASYNCHRONOUS distributed system, it is IMPOSSIBLE
to solve consensus if even ONE process can CRASH-FAIL.
Even with CRASH failures only (not Byzantine), with NO timing guarantees,
no deterministic protocol can simultaneously guarantee:
✓ Agreement (all decide same value)
✓ Validity
✓ Termination (protocol always terminates)
WHY: In an asynchronous system, you can't distinguish between a SLOW process
and a CRASHED process. A protocol that waits for all processes could wait forever
(if one crashed). A protocol that doesn't wait all might decide too early.
FLP does NOT mean consensus is impossible in practice. It means: in a fully asynchronous model, you must weaken one of: safety OR liveness OR the fault model.
Imagine a committee vote where any member might have left the room (crashed) without telling anyone. You can't tell if someone is "thinking slowly" or "has left permanently." If you wait for everyone to vote, you might wait forever (if someone left). If you stop waiting early, the person might come back with a different vote and upset the agreement. FLP says this is a fundamental impossibility — you can't solve it perfectly in a system with no timing guarantees.

Circumventing FLP — Practical Solutions

Since FLP kills deterministic consensus in pure asynchronous systems, real systems use these workarounds:

ApproachWhat it weakensExample
RandomizationTermination: probabilistic (not guaranteed but very likely)Ben-Or's randomized consensus, Ethereum's PoS
Partial synchronyAsynchrony assumption: assume "eventually synchronous"Paxos, Raft — have timeouts
Failure detectorsFault model: assume an unreliable "oracle" can suspect failuresChandra-Toueg failure detectors
Weaker consistencyAgreement: eventual consistency instead of strongDynamo, Cassandra

Randomized Distributed Agreement

🔮
2026 Prediction — HIGH (Part B Q7): "Discuss the randomized distributed agreement approach and its advantages" appeared as Part B Q7 in 2025 — a brand new topic in DS. New Part B topics persist for 2-3 years per RTU pattern (confirmed by DS DCE, Types of OS patterns). VERY HIGH for 2026 Part B.

Why Randomization?

Randomized algorithms break the FLP impossibility by sacrificing deterministic termination — they terminate with high probability rather than certainty. This is practically acceptable because the probability of non-termination can be made astronomically small.

Imagine a deadlocked meeting where no one agrees. Instead of a rule (which can be exploited by traitors), everyone secretly flips a coin and whispers the result to one trusted person. The randomness introduces "noise" that the adversary can't control — breaking the deadlock by accident. That's randomized consensus: use randomness to break ties that adversaries would otherwise exploit to prevent agreement.

Ben-Or's Randomized Algorithm (1983)

The classic randomized consensus algorithm that gets around FLP. Works even with Byzantine failures but achieves agreement with probability 1 (not certainty).

Ben-Or Algorithm — Round Structure
Each process maintains: value v (its current estimate), round r.
In each round, process executes two phases:
PHASE 1 (Report):
Broadcast (REPORT, r, v) to all processes
Wait for n-f REPORT messages for round r
If more than n/2 reports have the same value w:
→ update v = w (a near-consensus exists)
Else: v = ⊥ (no consensus yet)
PHASE 2 (Propose):
Broadcast (PROPOSE, r, v) to all processes
Wait for n-f PROPOSE messages for round r
If any PROPOSE has a specific value w (not ⊥):
→ v = w (adopt that value)
→ if more than n/2 proposed w: DECIDE(w) and output
Else: v = COIN_FLIP() ← RANDOMNESS HERE: flip a fair coin!
Start next round with new v
Why it works: The coin flip prevents adversary from consistently steering all processes to deadlock. Expected rounds to termination: O(2^n) worst case BUT in practice terminates in O(1) expected rounds. Used for: blockchain consensus, distributed databases under partial failure.

Advantages of Randomized Distributed Agreement

  • Circumvents FLP: Terminates with probability 1 — practically as good as guaranteed termination in real deployments
  • Simpler protocols: No need for complex failure detectors or partial synchrony assumptions
  • Byzantine resilient variants: With n ≥ 3f+1, can handle Byzantine failures probabilistically
  • Self-stabilizing: Protocol eventually terminates regardless of the start state
  • Practical performance: Expected O(1) rounds under reasonable adversary models

Limitations

  • Non-deterministic termination: Cannot guarantee exact termination time (though probability approaches 1 exponentially fast)
  • Need shared random coin: Some variants require a trusted shared randomness source (e.g., a threshold signature scheme)
  • Higher message complexity: Two phases per round, multiple rounds possible
🔗
Connection: FLP → Randomized Agreement

FLP proves deterministic consensus is impossible in asynchronous systems. Randomized agreement is the direct response — by adding a coin flip, the algorithm avoids the "adversary-controlled" scenarios FLP uses in its impossibility proof. The adversary can control process speeds but cannot control the outcome of a truly random coin flip.

Why Replicate Data?

Data replication means maintaining multiple identical copies of data on different machines. It is the primary mechanism for achieving fault tolerance and performance in distributed systems.

Imagine a very important document (your original birth certificate). If you store it in only one place, a fire can destroy it forever. So you make photocopies and keep them in different cities. If Mumbai is flooded, Delhi still has the copy. Replication is exactly this — keeping copies of data in multiple machines so that if one machine fails, data is still accessible from another. The tricky part: what if you update the document? You need to update ALL copies — keeping them consistent.
Three Goals of Replication
1. FAULT TOLERANCE: If one replica fails, others continue serving requests.
→ System availability increases. Single point of failure eliminated.
2. PERFORMANCE: Clients read from nearest replica — reduced latency.
→ Read load distributed across replicas (read scaling).
→ Write still goes to all replicas (write overhead).
3. SCALABILITY: More replicas → more read throughput without redesign.
Trade-off: Replication improves reads but makes writes harder (all replicas must be updated consistently). The more replicas, the harder consistency becomes.

Key Issues in Managing Replicated Data

These issues appeared directly in 2025 Part A Q10 and are expected in 2026:

  • Consistency: All replicas should show the same data. How quickly must they converge? Strong consistency (all agree immediately) vs. eventual consistency (converge over time).
  • Update Propagation: How are writes propagated to all replicas? Push (primary pushes to others) vs. Pull (replicas poll for updates) vs. Hybrid.
  • Replica Management: Where to place replicas? How many replicas are needed? Who decides which replica is the "primary"?
  • Concurrency Control: Multiple clients writing simultaneously — how to serialize conflicting writes across all replicas?
  • Failure Handling: What happens when a replica crashes? How to bring it back up-to-date (repair) without stopping service?

Replication Models

1. Active Replication (State Machine Replication)

Active Replication
All replicas are ACTIVE and process EVERY client request.
Client sends request to ALL replicas (or multicast).
All replicas execute the same operations in the same order.
All replicas produce the same result → client accepts first reply.
Advantages:
✓ No failover delay — any replica can serve at any time
✓ Handles crash failures transparently
✓ Can tolerate Byzantine failures with n ≥ 3f+1 replicas (PBFT)
Disadvantages (2024 Part A Q10):
✗ All replicas must process ALL requests — CPU overhead proportional to n
✗ Requires TOTAL ORDER delivery — all replicas must process in exact same order
✗ Expensive to ensure total order (atomic multicast needed)
✗ Non-deterministic operations problematic (e.g., random number generators)
Used in: PBFT, Hyperledger Fabric. Also called "State Machine Replication" (SMR). Total order = Atomic Multicast (next section).

2. Passive Replication (Primary-Backup)

Passive Replication (Primary-Backup)
ONE primary replica processes all client requests.
Primary updates state, then forwards update to backup replicas.
On primary failure: one backup is elected as new primary.
Two variants:
Cold Standby: Backup is offline. Long failover time (minutes).
Hot Standby: Backup is live and up-to-date. Fast failover (seconds).
Warm Standby: Backup updated periodically. Moderate failover.
Advantages:
✓ Simpler — only one node executes operations
✓ No total order requirement on replicas
✓ Works with non-deterministic operations
Disadvantages:
✗ Primary is bottleneck. Failover introduces downtime.
✗ Can only tolerate CRASH failures, not Byzantine failures
Used in: PostgreSQL streaming replication, MySQL replication, many cloud databases (RDS primary-standby).

Database Techniques for Replicated Data

When replication is applied to databases, specialized techniques ensure consistency:

  • 2-Phase Commit (2PC): Ensures atomic distributed transactions. Phase 1 (Prepare): coordinator asks all participants to prepare. Phase 2 (Commit/Abort): if all prepared, commit; else abort. Guarantees atomicity but blocks if coordinator crashes.
  • 3-Phase Commit (3PC): Adds a pre-commit phase between Prepare and Commit. Non-blocking but more messages. Rarely used in practice.
  • Quorum-Based Protocols: Read quorum Qr + Write quorum Qw > N (total replicas). Ensures every read sees the latest write. Used in Dynamo, Cassandra (tunable consistency).
  • Versioning: Each write creates a new version. Conflicts resolved by vector clocks (as in Amazon Dynamo). Multiple versions kept; application resolves conflicts.
Quorum Formula — Must Know
Qr + Qw > N (Read quorum + Write quorum > Total replicas)
Qw + Qw > N (Two concurrent writes cannot both succeed without overlap)
Example: N=5 replicas, Qr=3, Qw=3
Qr+Qw = 6 > 5 ✓ (every read will see latest write)
Qw+Qw = 6 > 5 ✓ (no two writes can proceed without one blocking the other)
Tuning: Qr=1, Qw=5 → very fast reads, slow writes
Qr=5, Qw=1 → slow reads, fast writes
Qr=3, Qw=3 → balanced (used by Cassandra default)
CAP Theorem: Choose any 2 of: Consistency, Availability, Partition tolerance. Distributed systems always have partitions → must choose C or A.

Atomic Multicast

What is Atomic Multicast?

Atomic Multicast (also called Total Order Broadcast or Atomic Broadcast) is a communication primitive that ensures all processes deliver the same set of messages in the exact same order. It is the foundation of active replication.

Imagine a class where the teacher announces: "Everyone write down the following announcement." Atomic Multicast is like having a loudspeaker that guarantees: (1) either EVERYONE in class hears the announcement or NO ONE does (atomicity), and (2) if two teachers speak at the same time, everyone hears them in the SAME order (total ordering). Normal gossip fails both — some people hear differently ordered messages.
Four Properties of Atomic Multicast
1. VALIDITY: If a correct process multicasts message m, then all correct processes
eventually deliver m. (No message gets lost)
2. UNIFORM AGREEMENT: If a process delivers m, then all correct processes
eventually deliver m. (All-or-nothing delivery)
3. UNIFORM INTEGRITY: m is delivered at most once by each process,
and only if it was multicast. (No duplicates, no forgeries)
4. TOTAL ORDER: If processes p and q both deliver messages m1 and m2,
then p delivers m1 before m2 iff q delivers m1 before m2.
(All processes see same ordering of all messages)
Total order is the hardest property. It requires consensus — which is equivalent to solving the consensus problem! This is why active replication is expensive.

Implementation Using Sequencer

One practical approach to implement atomic multicast is using a centralized sequencer:

Sequencer-Based Atomic Multicast
1. Process p wants to multicast message m.
2. p sends m to the SEQUENCER (a designated coordinator).
3. Sequencer assigns a SEQUENCE NUMBER to m → m gets global order position.
4. Sequencer broadcasts (m, seq_num) to all group members.
5. All processes deliver messages in sequence number order.
Guarantee: All processes deliver in same order ← total order ✓
Problem: Sequencer is single point of failure. Use backup sequencer for resilience.
Alternative: Lamport-clock based ordering — each process includes Lamport timestamp; all processes deliver in timestamp order (breaking ties by process ID). No single sequencer but O(n²) messages.

Update Propagation

Update Propagation Methods

Update propagation defines how changes made to one replica are spread to all other replicas. The choice of propagation method directly affects consistency, performance, and fault tolerance.

Three Propagation Approaches
1. PUSH (Eager) Protocol:
Primary replica proactively pushes updates to all replicas.
Replicas are always up-to-date. No staleness.
→ Good for: high read-to-write ratios, strong consistency
→ Bad for: many replicas (O(n) network on every write)
Example: MySQL synchronous replication
2. PULL (Lazy) Protocol:
Replicas periodically poll primary for updates.
Stale reads possible between polls. Weak consistency.
→ Good for: high write-to-read ratios, tolerance of staleness
→ Bad for: when freshness is critical
Example: DNS record TTL — replicas use cached records for hours
3. HYBRID (Lease-Based):
Primary gives replicas a "lease" (time-based right to cache data).
During lease: replica serves reads without contacting primary.
On write: primary waits for all leases to expire (or revokes them).
→ Balance: good read performance, consistent writes
Example: AFS file system callbacks — server notifies clients on change
Epidemic (Gossip) Protocol: Nodes randomly pick a neighbour and exchange state. Very scalable, eventual consistency. Used in Cassandra, Riak.

Epidemic / Gossip Protocols

A scalable approach to update propagation inspired by how rumours spread in a social network:

  • Each node periodically picks a random subset of nodes and exchanges its latest updates
  • Updates "infect" the network like a virus — spreading exponentially fast
  • Very scalable: O(log n) rounds to inform all n nodes
  • Fault tolerant: works even with many node failures
  • Consistency model: eventual consistency — all nodes eventually converge
  • Examples: Cassandra anti-entropy, Amazon Dynamo, Consul

What is CORBA?

CORBA (Common Object Request Broker Architecture) is a standard defined by the Object Management Group (OMG) that enables software objects written in different programming languages, running on different operating systems and hardware architectures, to communicate and interoperate seamlessly over a network.

Imagine a United Nations meeting where delegates speak French, English, Spanish, and Chinese. Without translators (middleware), they can't communicate. CORBA is the universal translator system — it lets a Java object running on Linux call a method on a C++ object running on Windows, as if they were in the same process. CORBA translates the "language" of each object so they understand each other across machine and language boundaries.
CORBA Key Concepts
CORBA Standard: Defines HOW distributed objects communicate, not HOW they work internally.
OMG: The standards body (like IEEE for networks). CORBA is their most famous standard.
ORB (Object Request Broker): The middleware bus — the HEART of CORBA. Routes calls between objects.
IDL (Interface Definition Language): Language-neutral way to define object interfaces.
IOR (Interoperable Object Reference): Unique handle to locate any CORBA object anywhere.
CORBA enables: Language interoperability (Java/C++/Python), OS interoperability (Linux/Windows), Hardware interoperability (x86/ARM), Location transparency.

CORBA Architecture

The Object Request Broker (ORB) — Core of CORBA

The ORB acts as a software bus connecting client objects and server objects. It handles: finding the server object, sending the request, marshalling (converting) parameters, and routing the reply. Neither client nor server knows about the network — only the ORB does.

Fig: CORBA Architecture — ORB is the central communication bus. Client uses Stub (or DII); Server uses Skeleton (or DSI). IDL compiler auto-generates both stubs and skeletons from the IDL specification.

Component-by-Component Explanation

CORBA Components — 8 Key Parts
1. ORB (Object Request Broker):
The middleware bus. Finds server objects, routes requests, handles marshalling.
Client and server NEVER communicate directly — always through ORB.
2. IDL (Interface Definition Language):
Language-neutral specification of object interfaces.
IDL Compiler generates: Client Stubs + Server Skeletons (for any target language)
3. Client Stub (Proxy):
Generated by IDL compiler. Makes remote calls look like local calls to client.
Marshals (packs) parameters into IIOP format; sends to ORB.
4. Server Skeleton:
Generated by IDL compiler. Receives request from ORB; dispatches to actual object.
Unmarshals parameters back into native language types.
5. DII (Dynamic Invocation Interface):
Client can call objects without a pre-compiled stub — discovers interface at runtime.
Slower but flexible (useful for scripting, testing, late binding).
6. DSI (Dynamic Skeleton Interface):
Server can handle requests without a pre-compiled skeleton — dispatches dynamically.
Enables bridges between ORBs, generic servers (ORB gateways).
7. Object Adapter (POA — Portable Object Adapter):
Connects server object to ORB. Manages object lifecycle: creation, activation, deactivation.
POA is the newer, portable version (older: BOA — Basic Object Adapter).
8. IOR (Interoperable Object Reference):
Globally unique ID for each CORBA object. Contains: host, port, object key, type info.
Client holds IOR to reference a remote object (like a pointer but network-wide).
IIOP (Internet Inter-ORB Protocol): TCP/IP-based wire protocol for CORBA. Ensures different vendors' ORBs can communicate. Runs on port 683 by default.

CORBA Invocation Flow — Step by Step

Request Flow: Client → Server (via CORBA)
Step 1: Client holds an IOR (object reference) for the remote server object.
Step 2: Client calls method on its local Stub: bankAccount.deposit(100.0)
Step 3: Stub MARSHALS (serializes) parameters into IIOP binary format.
Step 4: Client's ORB opens TCP connection to server's host:port (from IOR).
Step 5: ORB sends IIOP GIOP (General Inter-ORB Protocol) request message.
Step 6: Server ORB receives request, locates object via Object Adapter (POA).
Step 7: Server Skeleton UNMARSHALS parameters back to C++/Java/Python native types.
Step 8: Skeleton calls the actual server object method: deposit(100.0)
Step 9: Return value marshalled, sent back to client ORB.
Step 10: Client Stub unmarshals return value → client code sees local return value.
The client programmer wrote "bankAccount.deposit(100.0)" — looks EXACTLY like a local call. All 10 steps happened transparently. This is CORBA's key promise: location transparency.

CORBA RMI & Interface Definition Language (IDL)

CORBA IDL — The Universal Interface Language

IDL (Interface Definition Language) is the contract language of CORBA. You write the interface once in IDL and the IDL compiler generates stubs and skeletons for any target language (C++, Java, Python, Ada, COBOL, etc.).

IDL is like a restaurant menu. The menu (IDL) describes WHAT dishes are available and WHAT ingredients each requires — but not HOW the chef (the actual object) cooks them. Any customer (client in any language) can read the menu and order. Any restaurant (server in any language) can use the menu to know what to serve. The menu is the common contract — independent of who's cooking or who's eating.
IDL Syntax — Key Constructs
// Define a module (like a Java package / C++ namespace)
module BankingSystem {
// Define a custom exception
exception InsufficientFunds {
float balance; // How much was available
};
// Define the interface (like a Java interface)
interface BankAccount {
// Read-only attribute (getter only)
readonly attribute float currentBalance;
// Read-write attribute
attribute string accountHolder;
// Operations (methods)
float balance(); // No parameters, returns float
void deposit(in float amount); // in = input parameter
void withdraw(in float amount) // in = input, can raise exception
raises(InsufficientFunds);
void transfer(in float amount, // Multiple parameters
in BankAccount target)
raises(InsufficientFunds);
};
};
Parameter modes: in (client→server), out (server→client), inout (both directions). IDL compiler generates: BankAccount.java (client stub) + BankAccount.h+.cpp (server skeleton) from this one file.

How IDL Enables Interoperability

The IDL → Multiple Languages workflow is CORBA's core innovation:

IDL → Multiple Language Stubs (One Source, Many Targets)
ONE IDL file: BankAccount.idl
↓
IDL Compiler (e.g., omniidl, JacORB)
↓ generates:
Java Client Stub: BankAccountStub.java (Android app uses this)
C++ Skeleton: BankAccount_impl.h (Server in C++ uses this)
Python Stub: BankAccount_skel.py (Python script uses this)
Ada Skeleton: BankAccount.ads (Legacy military system uses this)
Result: Java client on Android calls C++ server on Windows!
ORB handles all translation transparently.
This is why CORBA is used in: military systems (Ada), finance (COBOL+Java), telecom (C++), and enterprise (Java). All interoperating through CORBA IDL.

1. Client obtains IOR of server object (from Naming Service or file)
2. Client narrows IOR to specific interface type: BankAccount ba = BankAccountHelper.narrow(obj)

8. Client receives return value — looks like a normal function call result

Connection-less Communication (UDP Sockets):

In connection-less communication, client sends a datagram directly to server without establishing a prior connection. No handshake, no state maintained.

UDP Socket Steps:
Server:
1. socket(AF_INET, SOCK_DGRAM, 0) — create UDP socket
2. bind(socket, port) — bind to port

4. sendto(response, client_addr) — reply to sender

Client:
1. socket(AF_INET, SOCK_DGRAM, 0) — create UDP socket
2. sendto(message, server_addr) — send directly (no connect step)

· No connection overhead (fast for short messages)
· No guaranteed delivery (packet may be lost)
· No ordering guarantee
· Good for: DNS queries, DHCP, video streaming, gaming (latency > reliability)

Distributed Shared Memory (DSM): Abstraction providing a shared address space across multiple physically separate machines. No hardware shared memory — software creates the illusion using virtual memory mechanisms.

Mechanism: When Process on Machine A accesses an address located on Machine B → hardware generates a page fault → OS intercepts → DSM software fetches the page from Machine B over network → maps it into A's address space → process continues without knowing the page was remote.

Architecture:
[Machine A: Process + Virtual Memory + DSM Software] ←→ Network ←→ [Machine B: Memory Pages + DSM Software]

Coherence Protocols:
· Write-Invalidate: On write, invalidate all remote copies → next read fetches fresh (used by Ivy)
· Write-Update: On write, push update to all copies → always fresh but high bandwidth
· Release Consistency: Sync only at lock acquire/release → used by TreadMarks, most practical

📋 Part B — 4 marks (Q7)
Unit 6 | Randomized Agreement
Discuss the randomized distributed agreement approach. How does it overcome the FLP impossibility result?
Randomized distributed agreement appeared as Part B Q7 in 2025 — a brand new anchor. RTU retains new Part B anchors for 2-3 years (confirmed by Types of OS 2025→2026, DCE 2024+2025). VERY HIGH for 2026 Part B Q7 — same slot, same topic.
VERY HIGH

FLP Impossibility: In an asynchronous system, deterministic consensus is impossible if even one process can crash (Fischer-Lynch-Paterson, 1985). Cannot distinguish slow process from crashed.

Randomized Agreement (Ben-Or Algorithm): Overcomes FLP by introducing a random coin flip — termination becomes probabilistic (not guaranteed) but occurs with probability 1.

Ben-Or — Each Round (2 Phases)
Phase 1 (Report): Broadcast current value v. Wait for n-f reports.
If majority has same value w → set v=w
Phase 2 (Propose): Broadcast v. Wait for n-f proposals.
If majority proposes w → decide w (output)
Else → v = COIN_FLIP() ← breaks adversary's control
Why it works: Adversary can control process speeds but NOT a random coin. The coin breaks deadlock that the adversary would exploit in FLP.

Advantages: No failure detectors needed, no timing assumptions, handles Byzantine failures, expected O(1) rounds in practice.

Byzantine Agreement escalated: Part B Q7 in 2024 → Part C Q5 in 2025. This is the clearest escalation pattern in DS history. Part C topics stay in Part C once they enter (confirmed by DSM — Part C all 3 years). CERTAIN for 2026 Part C.
CERTAIN — Part C 2026

Byzantine Agreement: n processes agree on a value despite f Byzantine-faulty processes. Requirements: Agreement, Validity, Termination. Key theorem: n ≥ 3f+1.

Approach 1 — Oral Messages (OM):

OM(0): Commander sends value; lieutenants use what they receive.
OM(m): Commander sends to all. Each lieutenant runs OM(m-1) with n-2 others.
Each lieutenant decides by MAJORITY of n-1 values collected.
Example n=4, f=1: L1 gets {attack,attack,retreat}→majority=attack ✓
Complexity: f+1 rounds, n≥3f+1, O(n^(f+1)) messages.

Approach 2 — Signed Messages:

Digital signatures prevent message forgery.
Traitor can't fake a loyal general's signature.
Needs only n ≥ f+2 (vs 3f+1 for oral messages).
Trade-off: Requires public-key infrastructure.

Approach 3 — PBFT (3-Phase Protocol):

① PRE-PREPARE: Primary assigns seq# + broadcasts to all replicas
② PREPARE: Each backup validates + sends PREPARE to all
→ "Prepared" = PRE-PREPARE + 2f PREPARE messages received
③ COMMIT: Each prepared replica sends COMMIT to all
→ Execute + Reply to client when 2f+1 COMMIT received
Message complexity: O(n²). Used in Hyperledger, Tendermint.

Fault Tolerance: Active replication (all process every request) + passive (primary-backup). Replication, checkpointing, message logging are the three recovery mechanisms.

Replication Goals: Fault tolerance (survive node failures), Performance (read from nearest replica), Scalability (add replicas for more read throughput).

Key Issues: Consistency (all replicas same data), Update Propagation (how writes spread), Concurrency control (simultaneous writes), Failure recovery (repair crashed replica), Total ordering (all replicas process in same order).

Active Replication (State Machine Replication):

All replicas process EVERY request. Client multicasts to all.
✓ No failover delay ✓ Tolerates Byzantine with n≥3f+1
✗ All replicas do all work (CPU overhead proportional to n)
✗ Requires TOTAL ORDER delivery (Atomic Multicast)
✗ Non-deterministic operations problematic

Passive Replication (Primary-Backup):

ONE primary processes requests; backups standby hot/warm/cold.
✓ Simple ✓ No total order requirement ✓ Works with non-determinism
✗ Primary is bottleneck ✗ Only crash-fault tolerance

Atomic Multicast (Total Order Broadcast): Ensures all replicas deliver the same messages in the same order. Properties: Validity, Uniform Agreement, Integrity, Total Order. Required by Active Replication. Implemented via sequencer or Lamport timestamp ordering.

Quorum: Qr + Qw > N ensures every read sees latest write
N=5, Qr=3, Qw=3: 6>5 ✓ — Cassandra default "quorum" setting
