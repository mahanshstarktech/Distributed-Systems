What is a Distributed System?

A Distributed System is a collection of autonomous computers linked by a network and equipped with distributed system software that enables the computers to coordinate their activities and share resources — hardware, software, and data — such that users perceive the system as a single integrated computing facility.

— Definition by Tanenbaum & Van Steen (the standard RTU answer)

Imagine a team of workers in different cities all working on one project. Each person has their own computer and internet connection. They don't share one giant computer — they are all independent. But they communicate and coordinate so well that to the client, it looks like ONE big office is handling everything. That's a distributed system. Your phone, the internet server, and the database server are all separate computers working together to show you Instagram — that's a distributed system in action.
Key Properties (Write in Exam — 5 Points)
1. Multiple autonomous computers (each with its own CPU, memory, OS)
2. Connected via a communication network (LAN, WAN, Internet)
3. No shared memory between nodes (message passing only)
4. No common physical clock (clocks may drift)
5. Appears as a single coherent system to users
Memory: "MNCNA" — Multiple computers, Network, No shared memory, No common clock, Appears single

Goals of a Distributed System

The four primary goals of building a distributed system are captured in the acronym ROST:

  • Resource Sharing: Hardware (printers, disks), software (files, databases), and data accessible across the network. Example: Cloud storage like Google Drive — files stored on remote servers but feel local.
  • Openness: Offer services according to standard rules (protocols) so components can easily be extended or replaced by others. Example: REST APIs allow any client (Android, iOS, Web) to communicate with one server.
  • Scalability: The system should remain effective as the number of users, processors, and geographic distance increases. Scalable in size (more users), geography (more locations), and administration.
  • Transparency: Hide the complexity of distribution from users and programmers. The system should feel like one unified machine even though it's many.

Nodes of a Distributed System

Each computer (node) in a distributed system can play different roles:

  • Client Node: Requests services from servers. Example: Your browser requesting a webpage.
  • Server Node: Provides services to clients. Example: Apache web server, database server.
  • Peer Node: Acts as both client and server simultaneously. Example: BitTorrent node — downloads from peers (client) while uploading to others (server).
  • Worker/Compute Node: Executes tasks in a distributed computation framework. Example: Hadoop DataNode running MapReduce tasks.
Fig: A Distributed System — multiple autonomous nodes connected via a network. Each node is completely independent yet they cooperate.

Features of Distributed Systems

Features vs Challenges — Two Sides of the Same Coin

Every feature of a distributed system comes with a corresponding challenge. Understanding this duality is key to answering exam questions that ask about both.

Think of building a restaurant chain. The benefit is you can serve more customers (scalability). But the challenge is: how do you keep all branches consistent? How do you handle a fire in one branch without shutting everything down? Features = the benefits of going distributed. Challenges = the headaches that come with it.

Distributed Computation Paradigms

What are Computation Paradigms?

A computation paradigm is a style or pattern of organizing distributed computation — how tasks are divided, how processes communicate, and how results are collected. Different paradigms suit different problem types.

Just like there are different styles of teamwork (everyone does the same job vs. everyone does a specialized job vs. one person coordinates all others), there are different styles of distributed computation. The paradigm is the "work style" of your distributed system.
6 Key Distributed Computation Paradigms
1. CLIENT-SERVER (most common):
Clients request; servers respond. Asymmetric roles. Stateless or stateful.
Example: Web browser (client) ↔ Apache web server
2. PEER-TO-PEER (P2P):
All nodes are equal — each acts as both client and server.
No central authority. Example: BitTorrent, blockchain
3. MESSAGE PASSING:
Processes communicate via explicit send/receive primitives.
Foundation of all distributed communication. Example: MPI (Message Passing Interface)
4. SHARED MEMORY (virtual):
Software creates illusion of shared memory across nodes (DSM).
Familiar programming model. Example: TreadMarks DSM system
5. DATAFLOW:
Computation triggered when all input data is available.
Good for stream processing. Example: Apache Spark streaming
6. MOBILE AGENTS:
Code migrates to data rather than data to code.
Agent travels to nodes, executes, returns result. Example: web crawlers
Most exam questions focus on Client-Server and P2P. Message Passing connects to Unit 3.

Models of Distributed Systems

Why Do We Need Models?

A model is an abstraction that helps us reason about a distributed system's behavior, especially regarding failures, timing, and communication. RTU syllabus focuses on three key models.

Three Models of Distributed Systems
1. INTERACTION MODEL (Timing/Communication):
Defines assumptions about message delivery time and process speed.
→ Synchronous Model: Known bounds on time (message delivery, process steps)
Easier to reason about; can detect failures by timeout
→ Asynchronous Model: No timing bounds (the internet is asynchronous!)
More realistic but harder — can't use timeouts to detect failures
2. FAILURE MODEL:
Defines what types of failures can occur.
→ Crash-stop: Process crashes and stops forever (simplest)
→ Omission: Process fails to send/receive messages
→ Byzantine: Process behaves arbitrarily (worst case — Unit 6)
3. SECURITY MODEL:
Defines threats: unauthorized access, eavesdropping, Trojan horses.
Assumes an adversary may intercept or modify messages.
In exam: Synchronous vs Asynchronous is tested in Part B (2024 Q2: "Compare synchronous vs asynchronous execution")

Types of Operating Systems

🔮
2026 Prediction — HIGH (Part B Q1): "Discuss different types of operating systems used in distributed system" appeared as Part B Q1 in 2025 — a new anchor question. RTU's pattern confirms new Part B anchors persist for 2-3 years (confirmed by GAE in CC Unit 3). The 2023 paper tested NOS vs DOS in Part A Q2. VERY HIGH confidence for 2026.

Overview

There are four types of operating systems relevant to distributed systems, evolving from centralized to fully distributed. Understanding the difference between Network OS and Distributed OS is a common exam trap.

Imagine how an office can be organized: (1) One computer, everyone sits at terminals (centralized). (2) Everyone has their own PC but shares files over network drives (Network OS — you know it's on a remote machine). (3) A magical office where data appears from wherever it is, you can't tell what's local or remote (Distributed OS — transparent). (4) Many expert departments that work independently but share info when needed (Cooperative Autonomous).

Type 1: Centralized OS

A Centralized OS runs on a single processor (or tightly coupled multiprocessors) and manages all resources on that one machine. All processing happens centrally; users access via terminals.

  • Example: Early mainframe systems — IBM System/360. Users connect via dumb terminals.
  • Advantage: Simple management, consistent view of all resources.
  • Disadvantage: Single point of failure. Cannot scale. Not suitable for distributed applications.
User Terminals → Mainframe (Single CPU) → All Resources
Think: One brain, many arms. Modern equivalent: A powerful AWS EC2 instance that all users SSH into.

Type 2: Network Operating System (NOS)

A Network Operating System provides networking capabilities on top of individual machines. Each machine runs its own standalone OS; the NOS adds utilities for file sharing, printer sharing, and remote login. Users are aware of the network — they explicitly connect to specific machines.

  • Example: Windows Server with file shares, Novell NetWare, early Unix with NFS.
  • Key Distinction: NOT transparent. User must know machine names (e.g., \\FileServer\SharedDocs).
  • Services: File sharing, shared printing, user authentication (Active Directory), email.
  • Advantage: Easy to understand and manage. Individual machines still work standalone.
  • Disadvantage: No transparency. Moving files changes addresses. User must manage location.

Type 3: Distributed Operating System (DOS)

A Distributed OS provides a single, coherent view of all resources across all connected machines. Users and programs interact with a single OS interface — the distribution is completely hidden (transparent). The OS manages all nodes as if they were one computer.

  • Example: Amoeba OS (Tanenbaum's research OS), Plan 9 from Bell Labs, Sprite, V-System.
  • Key Distinction: Full transparency. User never sees different machines — just one system.
  • Components needed: Process migration, global naming, distributed memory management, global scheduling.
  • Advantage: Highest transparency and ease of use. Best resource utilization.
  • Disadvantage: Extremely complex to build. Few production systems achieve true DOS.

Type 4: Cooperative Autonomous Systems

Cooperative Autonomous Systems are collections of independently managed systems that cooperate to accomplish tasks. Each system retains its own autonomy and administration but shares services with others when needed.

  • Example: The Internet itself — independent ISPs, universities, and companies cooperating via standard protocols. Grid computing (SETI@home). Service-oriented architectures (SOA) where independent microservices cooperate.
  • Key feature: No central authority. Cooperation is voluntary and standards-based.
  • Advantage: Scales globally. Each system maintains independence.
  • Disadvantage: Harder to guarantee Quality of Service (QoS) across autonomous systems.

Design Issues in Distributed OS

Overview

Building a distributed OS involves solving fundamentally harder problems than a centralized OS. These are the core design challenges every distributed system designer faces — and every RTU examiner expects you to know.

Building a distributed OS is like managing a company with offices in 100 different cities with no phone or internet (just postal mail). You need to figure out: How do departments communicate? How do you give things a name that works everywhere? How do you handle an office burning down? How do you make sure two offices don't both sell the same item? These are the design issues.
8 Core Design Issues (RTU expects all 8)
1. NAMING AND TRANSPARENCY:
How to identify resources (files, processes, nodes) uniquely across the system?
Challenge: Same name must work regardless of location (location transparency)
Solution: Distributed name service (DNS, LDAP, CORBA Naming Service)
2. COMMUNICATION:
No shared memory — all interaction via message passing over network
Challenge: Messages can be lost, delayed, duplicated, or reordered
Solution: Reliable transport protocols, RPC, message queues
3. FAULT TOLERANCE AND RELIABILITY:
Partial failures possible — how does system continue and recover?
Challenge: Can't tell if remote node crashed or is just slow (asynchrony)
Solution: Replication, checkpointing, distributed agreement protocols
4. SYNCHRONIZATION AND CONCURRENCY CONTROL:
Multiple processes access shared resources simultaneously
Challenge: Race conditions, deadlocks in distributed setting
Solution: Distributed locks, timestamps, optimistic concurrency control
5. SCALABILITY:
System must handle 10x users without redesign
Challenge: Central bottlenecks, global data structures
Solution: Decentralized algorithms, hierarchical structures
6. SECURITY:
Messages pass through untrusted networks — eavesdropping, forgery
Challenge: Authentication without shared secret; secure communication
Solution: Public-key cryptography, SSL/TLS, Kerberos
7. HETEROGENEITY:
Different hardware, OS, languages, data representations
Challenge: A Java int (big-endian) differs from a C int (little-endian)
Solution: Middleware (CORBA, JEE), standardized data formats (XML, JSON)
8. PROCESS MANAGEMENT AND SCHEDULING:
How to distribute processes across nodes for load balancing?
Challenge: Dynamic workload, process migration cost
Solution: Load balancing algorithms (sender-initiated, receiver-initiated)
RTU exam tip: "Design issues" questions — mention all 8 with a 1-line explanation each to fill pages.
🔗
Connection: Design Issues → Entire Syllabus

Every remaining unit addresses one design issue: Communication → Unit 3 (RPC, IPC). Fault Tolerance → Unit 6 (Byzantine). Synchronization → Units 3,5 (Logical Clocks, DME). Process Management → Unit 4 (Scheduling). Naming → Unit 4 (DFS). Security → Cross-cutting. This is why design issues are foundational!

Goals, Transparency & Services

Transparency in Distributed Systems — The 8 Types (EXAM CRITICAL)

Transparency is the concept of hiding the implementation details of the distributed system from users and applications. The ANSA/ISO Reference Model identifies 8 forms of transparency — RTU commonly asks for 5-6 of these.

Transparency in DS is like a great hotel concierge. You just say "I need a taxi" — you don't know which taxi company they call, whether they use an app or phone, or where the taxi is. The complexity is hidden from you. Each type of transparency hides a different kind of complexity from the user.
8 Types of Transparency — ALRMFCPP
1. ACCESS Transparency: Hide differences in data representation and invocation
Example: Access a Unix file and a Windows file using same API
2. LOCATION Transparency: Resource can be accessed without knowing physical location
Example: www.google.com — same URL regardless of which data centre responds
3. REPLICATION Transparency: Multiple copies of resource exist; user sees one
Example: DNS is replicated worldwide; you never ask "which DNS copy?"
4. MIGRATION (Relocation) Transparency: Resource can move without user noticing
Example: Google migrates your file between data centres — URL unchanged
5. FAILURE Transparency: System continues despite failures; user unaware
Example: Netflix continues streaming when one server fails (CDN reroutes)
6. CONCURRENCY Transparency: Multiple users access same resource; each feels alone
Example: 1000 users edit different Google Sheets; no user sees others' lock waits
7. PERFORMANCE Transparency: System can be reconfigured to improve performance
Example: Cloud auto-scaling — 10x load handled without user noticing delay
8. PERSISTENCE Transparency: Volatile/non-volatile nature of resource hidden
Example: Object database — object saved/loaded transparently without File I/O
Memory trick: "ALRMFCPP" — Access, Location, Replication, Migration, Failure, Concurrency, Performance, Persistence

Exam Trick — Most Important Transparencies (Pick 5 if 4-mark)

RTU's most frequently tested: Location, Replication, Failure, Migration, Concurrency — these 5 cover all exam scenarios. For a 2-mark answer: "Key forms: location (no need to know where), replication (multiple copies hidden), failure (continues despite crashes), migration (resource can move), concurrency (multiple simultaneous users)."

Scalability has three dimensions and corresponding solutions:

Services in Distributed Systems

Services are the fundamental units of functionality provided in a distributed system. A service has a well-defined interface — the service user doesn't need to know how the service is implemented.

Key Service Categories
Naming Service: Translates names to addresses (like DNS)
Time Service: Provides synchronized time across nodes (NTP)
Transaction Service: ACID guarantees for distributed operations
Security Service: Authentication, authorization, encryption
Directory Service: Look up users, resources, services (LDAP)
File Service: Distributed file access (NFS, AFS)
Service + Standard Interface = Service-Oriented Architecture (SOA) — the foundation of cloud computing

What are Architecture Models?

An architecture model defines how the components of a distributed system are organized, how they interact with each other, and where they are placed. It provides a blueprint for building the system.

Architecture models are like city planning blueprints. They decide: Will the city have one giant central hub (Client-Server), or lots of equal neighborhoods (P2P)? Will there be specialized districts (multi-tier)? The blueprint doesn't build the city — it defines the structure. In distributed systems, the architecture model defines which machines talk to which, who has power, and who does what.

1. Client-Server Architecture

The most common model. Clients make requests; servers provide responses. Roles are fixed and asymmetric.

Fig: Client-Server — asymmetric, well-understood, but server is a single point of failure
  • Advantages: Simple, well-understood, easy to secure (control at server), centralized management.
  • Disadvantages: Server is a bottleneck and single point of failure. Does not scale well.
  • Example: Web (browser ↔ Apache server), Email (Outlook ↔ Exchange), FTP.

2. Peer-to-Peer (P2P) Architecture

All nodes are peers — equal in role. Each node can act as both client and server. No central coordinator.

  • Advantages: No single point of failure. Scales naturally. Fault tolerant.
  • Disadvantages: Hard to manage, secure, and control quality. Finding resources is complex.
  • Example: BitTorrent, blockchain (Bitcoin), Napster (original music sharing), Gnutella.

3. Multi-Tier (N-Tier) Architecture

Extends client-server by separating functionality into multiple layers, each deployed on separate machines. The classic is 3-tier:

Fig: 3-Tier Architecture — Presentation, Business Logic, and Data layers independently deployable and scalable
  • Advantages: Each tier can be scaled independently. Separation of concerns — change business logic without touching DB or UI. Better security.
  • Disadvantages: More complex deployment. Network latency between tiers.
  • Example: Every major web application — Amazon, Flipkart, banking portals.

4. Service-Oriented Architecture (SOA)

Applications built from loosely coupled, independently deployable services communicating via standard protocols (REST, SOAP). Each service has a well-defined interface.

  • Advantages: Reusability, interoperability, independent updates.
  • Disadvantages: Service discovery overhead, governance complexity.
  • Example: AWS offers 200+ independent services; each is a separate SOA service.

What is DCE?

Distributed Computing Environment (DCE) is a set of software technologies developed by the Open Software Foundation (OSF) that provides a comprehensive framework for building distributed applications that work across heterogeneous hardware, OS, and network environments. DCE enables software running on different machines from different vendors to work together seamlessly.

Imagine you have workers using Windows, Mac, and Linux computers in different cities. They need to communicate and share files seamlessly. DCE is like a universal translator and coordinator — it hides all the differences between operating systems and machines, and gives everyone a common language and set of tools. You write your application once using DCE services, and it works everywhere.

DCE Core Components (RTU 2025 Part B Q2)

DCE — 6 Core Services (D-REMOTE)
1. REMOTE PROCEDURE CALL (RPC):
The heart of DCE. Allows a program to call a procedure on a remote machine
transparently as if it were local. Based on the Sun RPC mechanism.
Components: Client stub → Network → Server stub → Actual procedure
2. DIRECTORY SERVICES (CDS — Cell Directory Service):
Naming service that maps logical names to network locations
Global Directory Service (GDS): for world-wide name resolution
Cell Directory Service (CDS): local naming within a DCE cell
3. DISTRIBUTED FILE SYSTEM (DFS):
Provides access to files on remote machines as if they were local
Based on AFS (Andrew File System) — supports caching, replication
4. DISTRIBUTED TIME SERVICE (DTS):
Synchronizes clocks across all DCE nodes to a global time source
Essential for ordering events and transaction timestamps
5. SECURITY SERVICE:
Authentication using Kerberos protocol (ticket-based)
Authorization: Access Control Lists (ACLs)
Encryption: DES-based encryption of communications
6. THREADS SERVICE:
POSIX-compliant thread library for concurrent programming
Allows servers to handle multiple requests simultaneously
DCE "Cell" = administrative unit — a group of machines managed together. Can span multiple LANs.

Why is Time a Problem in Distributed Systems?

In a single computer, time is trivial — the OS has one clock, all processes use it. In a distributed system, each machine has its own clock and these clocks drift apart over time. There is no global clock. This creates a fundamental problem: we can't use timestamps to determine the order of events.

Imagine you and your friend in different cities are running a relay race. You each have different stopwatches. Your watch says you started at 10:00:01 and your friend's says they received the baton at 10:00:00. Who went first? It looks like your friend started before you passed the baton — impossible! The clocks are out of sync. Distributed systems have this exact problem, and Lamport clocks are the solution.

The "Happened-Before" Relation (→)

Happened-before (denoted →) is Lamport's key insight for ordering events without physical clocks. It captures causality:

Happened-Before (→) — Definition
A → B (A happened before B) if ANY of these is true:
Rule 1: A and B are in the same process, and A occurs before B
Rule 2: A is the sending of a message and B is the receiving of that message
Rule 3: If A → C and C → B, then A → B (transitivity)
CONCURRENT events: If neither A→B nor B→A, then A and B are CONCURRENT (||)
Concurrent means: neither causally influences the other
Key insight: "A happened before B" means A COULD HAVE CAUSED B. It doesn't mean A physically occurred before B on a wall clock.

Lamport's Logical Clock Algorithm (2023 Part A Q5, 2023 Part C Q4)

Each process Pi maintains a counter Ci (initially 0):

Lamport Clock Rules (Memorize All 3)
LC1: Before executing any event, Pi increments its counter: Ci = Ci + 1
LC2: When Pi sends a message m, it sets timestamp t = Ci and sends (m, t)
LC3: When Pj receives (m, t):
Cj = max(Cj, t) + 1
[Take max of local counter and received timestamp, then increment]
Key property: If A → B then C(A) < C(B). BUT: C(A) < C(B) does NOT imply A → B (they may be concurrent).
Fig: Lamport Clock Space-Time Diagram. Numbers in brackets = Lamport timestamps. Arrows = messages between processes.

Vector Clocks — Detecting Concurrency

Lamport clocks have a limitation: C(A) < C(B) does NOT mean A happened before B. Vector Clocks solve this by assigning a vector of n counters (one per process) to each event, enabling detection of concurrent events.

Vector Clock Rules (n processes)
Each process Pi maintains vector Vi[1..n], all initially zero.
VC1: Before event: Vi[i] = Vi[i] + 1
VC2: Send message: attach entire vector Vi
VC3: Receive message with vector Vj:
For each k: Vi[k] = max(Vi[k], Vj[k])
Then: Vi[i] = Vi[i] + 1
COMPARISON RULES:
Va = Vb iff Va[k] = Vb[k] for all k
Va < Vb iff Va[k] ≤ Vb[k] for ALL k, AND Va ≠ Vb → A happened before B
Va || Vb iff neither Va < Vb nor Vb < Va → A and B are CONCURRENT
Vector Clocks vs Lamport: Vector Clocks CAN detect concurrent events. Lamport Clocks CANNOT. Lamport: O(1) overhead. Vector: O(n) overhead per message.

Exam Trick — Lamport vs Vector in One Line

Vector Clock: Va<Vb iff A→B. Va||Vb iff concurrent. Detects concurrency. O(n) overhead.

Recording State of Distributed Systems

The Global State Problem

In a single computer, recording state is trivial — pause the OS, copy all memory and registers. In a distributed system, we cannot pause all processes simultaneously (no global clock, no shared memory). Yet we need to capture a consistent global state for:

  • Deadlock detection (is there a cycle in the wait-for graph?)
  • Termination detection (has the distributed computation finished?)
  • Checkpointing for fault tolerance
  • Garbage collection of distributed objects
Imagine photographing a busy highway at rush hour. You can't stop all cars simultaneously — you can only photograph one car at a time. Yet you want a snapshot showing the positions of ALL cars at "one moment." If car A has already moved past car B when you photograph B, your snapshot might be inconsistent. Chandy-Lamport's algorithm solves this — it finds a consistent global snapshot even without stopping all processes simultaneously.

Consistent Global State

A global state of a distributed system consists of the local states of each process plus the state of all communication channels (messages in transit). A global state is consistent if it could have occurred in some correct execution — no message was recorded as received without being recorded as sent.

Consistency Rule
For each message m:
If m is recorded as RECEIVED in the global snapshot,
then m MUST be recorded as SENT in the same snapshot.
(Can't receive what wasn't sent)
INCONSISTENT: P1 records "sent m" in snapshot, P2 records "already received m"
→ Message appears neither in transit nor received — it "disappeared"!
A consistent snapshot is also called a "cut" in the space-time diagram where all arrow heads are to the RIGHT of the cut line.

Chandy-Lamport Snapshot Algorithm

The most famous distributed snapshot algorithm. Uses MARKER messages to record state without stopping any processes.

Chandy-Lamport Algorithm — Steps
INITIATOR (any process Pi that wants to start a snapshot):
1. Record own local state (LSi)
2. Send a MARKER message on ALL outgoing channels
3. Start recording incoming messages on all incoming channels
ON RECEIVING A MARKER on channel Cij at process Pj:
Case A: Pj has NOT yet recorded its state:
→ Record Pj's local state (LSj)
→ Set channel state of Cij = EMPTY
→ Send MARKER on ALL of Pj's outgoing channels
→ Start recording messages on all other incoming channels
Case B: Pj HAS already recorded its state:
→ Channel state of Cij = all messages received on Cij SINCE Pj recorded state
TERMINATION: When every process has received MARKER on all incoming channels.
Global snapshot = ∪(local states) + ∪(channel states)
Key insight: MARKER acts as a "time boundary" — everything before MARKER is in the snapshot, everything after is not.
Fig: Chandy-Lamport Algorithm — P1 records state, sends MARKER. P2 records state when MARKER arrives. Channel state captured between snapshot and MARKER receipt.

Q3. Define Inter-process Communication.
2024 Part A Q3 · 2 marks ⭐⭐Unit 3IPC
Inter-process Communication (IPC) is a set of mechanisms that allow processes running on the same or different machines to communicate and synchronize their actions.

IPC Mechanisms in DS:
1. Message Passing: Explicit send/receive primitives. Synchronous (blocking) or asynchronous (non-blocking).
2. Remote Procedure Call (RPC): Makes remote calls look like local function calls.
3. Shared Memory: Processes share a region of memory (requires synchronization).
4. Pipes: Unidirectional byte stream between related processes.
5. Sockets: Network endpoints for client-server communication.

In distributed systems, message passing is the PRIMARY mechanism since no shared memory exists between nodes.
Q1(Part B). Briefly explain the disadvantages of distributed system.
2024 Part B Q1 · 4 marks ⭐⭐Unit 2DS Disadvantages
Disadvantages of Distributed Systems:

1. Complexity: Much harder to design, build, and manage than centralized systems. Concurrency control, naming, failure handling all need distributed solutions.
2. Security Risks: Data passes over untrusted networks — eavesdropping, man-in-the-middle attacks, unauthorized access.
3. Network Dependency: System performance depends on network — if network fails, the entire distributed system may become unavailable.
4. Consistency Problems: Maintaining consistent copies of data across multiple nodes is very difficult (CAP theorem).
5. Debugging Difficulty: Race conditions and timing-dependent bugs are nearly impossible to reproduce and debug.
6. Partial Failure: Some nodes fail while others continue — system may be in an inconsistent state that's hard to detect and recover.
7. Overhead: Message passing, serialization, network latency add overhead compared to local function calls.
Q4(Part B). Explain different kinds of problems that are associated with the coordination and agreement in distributed system.
2024 Part B Q4 · 4 marks ⭐⭐Units 2, 5, 6Coordination Problems
Coordination and Agreement Problems:

1. Leader Election: Processes must agree on a single coordinator. Problem: no shared memory, processes may fail. Algorithm: Bully algorithm, Ring algorithm.
2. Distributed Mutual Exclusion: Only one process at a time enters critical section. Problem: no shared semaphore. Solution: Ricart-Agrawala, Token Ring.
3. Distributed Deadlock: Cycle in distributed wait-for graph. Hard to detect — no single node sees the full graph. Solution: Chandy-Misra-Haas algorithm.
4. Distributed Termination: Has a distributed computation finished? Hard to know — messages may still be in transit. Solution: Dijkstra-Scholten algorithm.
5. Consensus Problem: All processes agree on a single value despite failures. Byzantine Agreement is hardest form. Impossibility result (FLP): impossible in asynchronous system with even one crash failure.
6. Distributed Snapshots: Record consistent global state without stopping system. Solution: Chandy-Lamport algorithm.
Q7. What is non uniform memory access model?
2023 Part A Q7 · 2 marks ⭐⭐Unit 5NUMA
NUMA (Non-Uniform Memory Access) is a computer memory architecture used in multiprocessor systems where the time to access memory depends on the memory location relative to the processor — memory access is non-uniform.

Why it exists: In SMP (Symmetric Multiprocessing), all CPUs share one memory bus — creates bottleneck. NUMA gives each CPU (or group) its own local memory bank — accessing local memory is FAST; accessing another CPU's memory is SLOW (via interconnect).

Key Characteristics:
· Local memory access: ~2-5 ns (fast)
· Remote memory access: ~10-40 ns (slow — goes over interconnect)
· OS is NUMA-aware: tries to allocate memory on the same node as the CPU using it

Example: Modern Intel/AMD multi-socket servers. 4-socket server = 4 NUMA nodes, each with local RAM.

What is Byzantine Agreement? Explain different approaches to solve it. Also discuss fault tolerance in distributed systems.

Byzantine Agreement: n processes must reach consensus even if up to f are Byzantine faulty (send arbitrary/conflicting messages).

Requirements: Agreement (all non-faulty agree on same value) · Validity · Termination.

Impossibility: Need at least 3f+1 processes to tolerate f Byzantine failures
n=4, f=1: 4 ≥ 3(1)+1 = 4 ✓ | n=3, f=1: Impossible

Approaches:

  • Oral Messages (Lamport-Shostak-Pease): Recursive; majority voting; O(n^(f+1)) messages; needs n≥3f+1
  • Signed Messages: Uses digital signatures; traitor can't forge loyal general's message; needs n≥f+2
  • PBFT: 3-phase (Pre-prepare → Prepare → Commit); O(n²) messages; used in Hyperledger, blockchain

Fault Tolerance: Replication (active: all replicas process every request; passive: primary + backup), checkpointing, message logging, recovery protocols.

HIGH confidence

DFS allows access to remote files as if local. Examples: NFS, AFS, Coda, HDFS.

Design Issues:

Naming: Location-dependent (//server/path) vs Location-independent (transparent)
Caching: Reduces server load. Consistency: write-through, callbacks (AFS), time-based (NFS)
Replication: Fault tolerance. Primary-backup, quorum-based, eventual consistency (Coda)
NFS vs AFS
NFS: Stateless · Server-side caching · Poll-based · ~100 clients
AFS: Stateful · Client whole-file caching · Callbacks · 50,000+ clients

Implementation: RPC for client-server comm, VFS (Virtual File System) layer in OS, consistency protocol as background daemon.

📖 Part C — 10 marks (Q5)
Unit 5 | Resource Management
With the help of suitable diagrams, explain sender-initiated and receiver-initiated algorithms for dynamic load sharing and balancing in distributed systems.
Load sharing algorithms appeared in 2023 Part C Q5. ABSENT from 2024 and 2025 — a 2-year gap. The 2-year absence with a confirmed 2-year rotation pattern in RTU (validated by 2026 CAO, DIP, ML) makes this VERY HIGH for 2026 Part C Q5.
VERY HIGH confidence (2-year absence)

Sender-Initiated: Overloaded node proactively finds lightly loaded node and sends work.

Steps: 1. Load > threshold → 2. Poll random nodes → 3. Transfer process to lightest
+ Effective at distributing overload | – Floods network when ALL nodes busy

Receiver-Initiated: Lightly loaded (idle) node actively requests work from overloaded nodes.

Steps: 1. Load < threshold → 2. Poll nodes for load → 3. Request process from busiest
+ Effective when system light | – Broadcast storm when ALL nodes idle

Symmetric: Combines both — above threshold acts as sender, below threshold acts as receiver. Best overall performance.

Summary: Sender-initiated → better for heavy load. Receiver-initiated → better for light load.
Symmetric → best for variable load. Real system: Condor (receiver-initiated).
