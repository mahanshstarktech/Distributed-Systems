Non-Uniform Memory Access (NUMA) Architecture

🔮
2026 Prediction — HIGH (Part A Q7): "What is non-uniform memory access model?" appeared in 2023 Part A Q7. A foundational question that connects directly to DSM (Unit 5's main topic). RTU pattern shows that hardware architecture questions alternate every 2 years. VERY HIGH for 2026.

What Problem Does NUMA Solve?

Before NUMA, multiprocessor systems used SMP (Symmetric Multiprocessing) — all CPUs share ONE memory bus. Imagine 16 CPUs all fighting to access the same memory bus. The bus becomes a massive bottleneck. NUMA solves this by giving each CPU (or group of CPUs) its own local memory bank, while still allowing access to other CPUs' memory when needed.

Imagine a library (memory) with one desk (memory bus). If 16 students (CPUs) all rush to the same desk, only one can be served at a time — everyone else waits. NUMA solution: Give each student their own mini-library near their desk. They can get books from their own shelf instantly, but if they need a book from another student's shelf, they have to walk there (slower, but still possible). Their "local" books are fast. "Remote" books take longer. That's NUMA — Non-Uniform Memory Access (speed is non-uniform depending on where the memory is).

NUMA — Key Concept

In a NUMA system, the time to access memory depends on where the memory is located relative to the processor:

  • Local Memory Access: Fast — memory physically close to the CPU (same node). Typical: 2–5 nanoseconds.
  • Remote Memory Access: Slow — memory on a different node, accessed via an interconnect (QPI, HyperTransport). Typical: 10–40 nanoseconds.
NUMA — Structure (Memorize This)
Node 0: [CPU 0, CPU 1] ←→ [Local Memory Bank 0]
Node 1: [CPU 2, CPU 3] ←→ [Local Memory Bank 1]
Node 2: [CPU 4, CPU 5] ←→ [Local Memory Bank 2]
↑ ↑
All nodes connected via HIGH-SPEED INTERCONNECT (QPI / HyperTransport)
CPU0 accessing Memory Bank 0 → LOCAL → FAST (~2ns)
CPU0 accessing Memory Bank 2 → REMOTE → SLOW (~20ns)
Key: Access time is NON-UNIFORM — depends on which node the memory is on.
Fig: NUMA 4-Node System — each node has local CPUs + local memory. Remote access goes through interconnect (slower).

Types of NUMA Systems

  • CC-NUMA (Cache Coherent NUMA): Hardware ensures all caches stay consistent automatically. CPUs share a single address space. Most modern multi-socket servers. Example: Intel Xeon E7 family.
  • NC-NUMA (Non-Cache Coherent NUMA): No hardware cache coherence — software must manage consistency. Used in very large systems (IBM SP2).
  • COMA (Cache-Only Memory Architecture): All memory is treated as cache — data migrates to where it's used. No fixed home node for data. Complex but efficient for irregular workloads.

Memory Consistency Models

🔮
2026 Prediction — CERTAIN (Part B Q5): "Explain the concept of memory consistency models. Why are they important?" appeared in 2025 Part B Q6 — a brand-new anchor question. RTU's confirmed pattern (CAO, DIP, ML) retains new Part B anchors for 2–3 years. CERTAIN for 2026.

Why Do We Need Consistency Models?

In a distributed or multiprocessor system, when Process P1 writes a value to shared memory, when does Process P2 (on a different machine) see that new value? The answer depends on the memory consistency model — a contract between the hardware/software and the programmer specifying what values a read operation can return.

Imagine you post something on Instagram. Your friends in different countries see it at different times — some instantly, some after a few seconds, some after minutes. What guarantee does Instagram give you? That's their "consistency model." In shared memory systems, the same problem exists: when you write to memory, when does everyone else see the new value? Different consistency models give different guarantees — stronger guarantees = more expensive, weaker = cheaper but harder to program.

The Consistency Spectrum

Consistency models range from Strict (strongest, most expensive) to Eventual (weakest, cheapest). The key trade-off: stronger consistency = easier programming but lower performance.

1. Strict Consistency (Strongest)

Any read always returns the value of the most recent write globally. If P1 writes X=5 at time t, any read of X after time t on any processor returns 5.

Strict Consistency Rule
Any read(X) returns the value written by the MOST RECENT write(X) globally.
Timeline: P1 writes X=5 at t=10. P2 reads X at t=11 → must get 5.
Requires GLOBAL CLOCK (impossible in DS!). Cannot be implemented in distributed systems.

2. Sequential Consistency (Lamport, 1979)

The most practically important strong model. Sequential Consistency requires two properties:

  1. All memory operations appear to execute in some sequential order (not necessarily real time).
  2. The operations of each individual process appear in program order (as written in the program).
Sequential Consistency Rules
Rule 1: Each process's operations execute in program order.
Rule 2: All processors see ALL operations in the SAME GLOBAL ORDER.
Example — VALID under Sequential Consistency:
P1: write(X,1) write(Y,1)
P2: read(Y)=1 read(X)=1 (Sees both writes, in some order)
INVALID example (violates SC):
P1: write(X,1) P2: write(Y,1)
P3: read(X)=1, read(Y)=0 | P4: read(Y)=1, read(X)=0
→ P3 and P4 see operations in DIFFERENT orders — not sequentially consistent!
Most intuitive model for programmers. Java/C++ memory model is based on it. Expensive in hardware.

3. Causal Consistency

Only causally related writes must be seen in the same order by all processes. Concurrent (non-causal) writes can be seen in different orders.

Causal Consistency Rule
If write A causally precedes write B (A→B), then:
ALL processes must see write A before write B.
BUT: concurrent writes (A || B) can be seen in any order.
Example:
P1: write(X,1)
P2: read(X)=1, write(Y,2) [P2's write causally follows P1's write]
P3: read(Y)=2, read(X)=1 [must see P1's write before P2's write]
P4: read(Y)=2, read(X)=0 [INVALID — must see X=1 if it sees Y=2]
Implemented using Vector Clocks (from Unit 2). Weaker than Sequential but still captures programming intent.

4. FIFO (PRAM) Consistency

Writes from a single process are seen by all other processes in the order they were issued. But writes from different processes may be seen in different orders by different processes.

P1: write(X,1), write(X,2), write(X,3)
P2 must see: X=1, then X=2, then X=3 (in order)
P3 can see them in same order: X=1, X=2, X=3
But if P4 also writes: P2 and P3 might see P1 and P4 writes interleaved differently.
Simple to implement. Used as a baseline. Does NOT require atomic broadcasting.

5. Release Consistency (Practical)

Memory is synchronized only at explicit synchronization points — acquire (lock) and release (unlock) operations. Between synchronization points, reads may return stale values.

Release Consistency Phases
acquire(lock) → SYNCHRONIZED START: see all writes up to previous release
[critical section: reads/writes may be non-coherent internally]
release(lock) → SYNCHRONIZED END: all writes visible to next acquirer
Between acquire and release: no guarantees about visibility to others
Used by TreadMarks DSM system. Very efficient — minimizes coherence traffic dramatically.

6. Eventual Consistency (Weakest)

If no new updates are made to a data item, all replicas will eventually converge to the same value. No timing guarantee — convergence may take seconds or minutes.

  • Used in: Amazon Dynamo, Apache Cassandra, DNS.
  • Perfect for: shopping carts (eventual sync is fine), DNS records (propagation delay acceptable).
  • Not for: bank transactions (you want immediate, correct balance).

Multiprocessor Cache Systems & Cache Coherence

The Cache Coherence Problem

In a multiprocessor system, each CPU has its own cache (L1, L2, L3). When multiple CPUs cache the same memory location and one CPU modifies it, the other caches have stale (outdated) copies. This is the cache coherence problem.

Imagine 4 students each have a photocopy of the same textbook page. One student (CPU) changes a word in their copy. Now the other 3 copies are wrong! If they all try to use their copies, they'll get different answers. The "cache coherence protocol" is the system for keeping all copies correct — either notify others of changes (write-update) or tell them their copy is now invalid (write-invalidate).
Fig: Cache Coherence Problem — CPU 1 updates X=10 in its cache, but CPUs 2 and 3 still see the old value X=5

Cache Coherence Protocols

Two main approaches to solving cache coherence:

Write-Invalidate Protocol (MSI, MESI — most common)
When CPU1 writes to X:
1. CPU1 broadcasts INVALIDATE(X) on the bus
2. All other caches mark their copy of X as INVALID
3. CPU1 writes new value to its cache
4. Next read by CPU2: cache miss → fetch fresh copy from CPU1 or memory
States (MESI protocol): M=Modified, E=Exclusive, S=Shared, I=Invalid
Write-Invalidate: One writer at a time. Low bus traffic for consecutive writes. Used by Intel x86.
Write-Update Protocol (Dragon protocol)
When CPU1 writes to X:
1. CPU1 broadcasts UPDATE(X, new_value) on the bus
2. All other caches UPDATE their copy to new value immediately
3. All caches stay valid but have latest value
Pros: Fewer cache misses after write | Cons: High bus traffic on every write
Better when data shared by many readers who read often after a write. Rarely used today.

What is DSM? The Big Picture

Distributed Shared Memory (DSM) is a software abstraction that provides the illusion of a single shared memory address space across multiple machines that do NOT physically share memory. Processes on different machines can read and write to shared variables as if they were accessing local memory — the DSM system handles all network communication transparently.

Imagine 4 people in different cities each with their own whiteboard (local memory). DSM creates a "virtual whiteboard" that all 4 people can see and write on — even though physically each person only has their own whiteboard. When person A writes "Hello" on the virtual whiteboard, DSM software makes sure persons B, C, and D eventually see "Hello" on their local whiteboards too. The trick: they're all actually working on their own copy, and DSM syncs changes between them.

Physical Reality vs DSM Abstraction

WITHOUT DSM (Physical Reality)
Machine A: Process P1 has Memory A (private)
Machine B: Process P2 has Memory B (private)
Communication: P1 must explicitly send(msg) and P2 must recv(msg)
→ Programmer must manage all communication explicitly
WITH DSM (Abstraction)
Machine A: Process P1 sees SHARED address space (0...N)
Machine B: Process P2 sees SAME SHARED address space (0...N)
Communication: P1 writes x[100]=5. P2 reads x[100] → gets 5.
→ DSM handles all network communication TRANSPARENTLY
DSM = Distributed System + Shared Memory Programming Model. Best of both worlds!

· No explicit message passing needed by programmer
· Network communication hidden by DSM layer
· Consistency maintained by coherence protocols (write-invalidate or write-update)

Q8. How does distributed shared memory improve system performance?

DSM improves system performance in the following ways:

1. Reduced Programming Overhead: Programmers use shared memory model — no need to partition data, write send/recv, or manage communication. Less development time = faster deployment of parallel applications.

Implementation of DSM Systems

🔮

Design Issues in DSM

Before implementing DSM, designers must decide four key issues:

Issue 1: Granularity

What is the unit of data transfer between nodes?

  • Page-based (most common): Transfer entire virtual memory pages (4 KB). Uses existing OS VM hardware. Simple but suffers from false sharing.
  • Word/byte-based: Transfer only the specific word/byte accessed. Minimal data transfer but very high overhead per access.
  • Object-based: Transfer shared objects. Only modified fields sent. Avoids false sharing. Used in object-oriented DSM (Linda, Orca).
False Sharing — THE Classic DSM Problem
Page contains: [variable A (used by P1)] [variable B (used by P2)]
P1 writes to A → entire page invalidated
P2 wants to read B → page fault! Must fetch entire page from P1
P2 writes to B → entire page invalidated again
P1 wants A → page fault again!
A and B are on the SAME PAGE but logically UNRELATED.
They're "falsely sharing" a page, causing massive coherence traffic (ping-pong effect).
Solution: Object-based DSM (only transfer the specific variable), or programmer restructures data layout.

Issue 2: Structure of Shared Address Space

  • Single address space: One flat virtual address space across all nodes. Simple for programmers. Example: Ivy.
  • Partitioned address space: Each node "owns" a portion; pages migrate on access. Example: TreadMarks.
  • Object-based space: Shared data structured as typed objects. Access via method calls. Example: Orca, Linda tuple spaces.

Issue 3: Scalability

As nodes increase, coherence protocol messages increase quadratically with write-update (every write notifies all nodes). Solutions:

  • Use write-invalidate (O(n) invalidations, not updates).
  • Hierarchical DSM — organize nodes in tree, propagate invalidations up/down.
  • Lazy release consistency — batch updates and sync only at lock boundaries.

Issue 4: Heterogeneity

Different machines may have different byte orderings (big-endian vs little-endian), word sizes (32-bit vs 64-bit), or data formats. DSM must convert data on transfer. Object-based DSM handles this naturally.

DSM Implementation Approaches

1. Page-Based DSM (Ivy — Li & Hudak 1986)
Mechanism: Uses OS virtual memory hardware (MMU + page tables)
→ Pages on other nodes marked INVALID in local page table
→ Access to invalid page triggers hardware page fault
→ OS calls DSM page fault handler
→ DSM sends page request to "manager" of that page
→ Page transferred over network → mapped in local address space
IVY coherence: Write-Invalidate. Single writer, multiple readers (SWMR).
On write request: invalidate ALL other copies → writer gets exclusive access
Ivy: Simple but high latency. Every write to a shared page causes network traffic.
2. TreadMarks — Lazy Release Consistency
Key Insight: Only sync memory at lock boundaries (acquire/release)
Between lock boundaries: writes are local only (no network traffic)
Protocol: Multiple-Writer Protocol
→ Multiple nodes can write the SAME page simultaneously
→ Each creates a "diff" (list of changes made)
→ At release: diffs are stored
→ At next acquire of same lock: diffs applied to page
Twin: Copy of page made before first write
Diff = XOR(twin, modified page) → compact representation of changes
TreadMarks dramatically reduces network traffic vs Ivy. False sharing still possible but less harmful.
3. Object-Based DSM (Orca, Munin)
Shared data structured as typed objects
Access only via object methods (no raw memory access)
When object accessed remotely: transfer only changed fields
Avoids false sharing completely (no pages — just objects)
Munin: "Multiple consistency protocol" — different consistency protocols
for different access patterns (write-shared, producer-consumer, etc.)
Most flexible but requires programmers to use object model. Used in research systems.

Q4(Part C). What are the design and implementation issues in distributed shared memory?
2024 Part C Q4 · 10 marks ⭐⭐⭐Unit 5DSM Design & Implementation🔮 CERTAIN 2026
Distributed Shared Memory provides illusion of shared memory across physically distributed nodes.

Design Issues:
1. Granularity: Unit of transfer — page (4KB, simple but false sharing), word (precise but overhead), object (flexible, avoids false sharing). Page-based most common using OS VM hardware.
2. False Sharing: Two unrelated variables on same page cause unnecessary invalidations — ping-pong effect. Solution: object-based DSM or careful data layout.
3. Scalability: Coherence messages grow with nodes. Use lazy protocols, hierarchical structures.
4. Heterogeneity: Different byte orders, word sizes — data conversion on transfer.

Implementation Approaches:
1. Page-Based (Ivy): Uses OS VM hardware. Invalid page → page fault → DSM fetches from owner. Write-invalidate: single writer, multiple readers. Simple but high latency for shared writable data.
2. TreadMarks (Lazy Release Consistency): Multiple writers per page. Creates "diffs" (XOR of before/after). Syncs diffs only at lock acquire/release. Dramatically reduces network traffic.

Models of Distributed Computation: Causality & Snapshots

Models of Distributed Computation — Overview

A model of distributed computation formalizes how we reason about what happened in a distributed system — which events occurred, in what order, and what the system state was at any point. These models are needed because there is no global clock and no global state visibility in real distributed systems.

Preliminaries — Events and Processes

A distributed computation consists of a set of processes, each executing a sequence of events:

Types of Events
Internal Event: computation within a process (no communication)
Send Event: process sends a message (always followed by a receive event somewhere)
Receive Event: process receives a message sent by another process
Process: pi = <ei1, ei2, ei3, ...> — ordered sequence of events
Distributed Computation: C = (E, →) — events E with happened-before ordering →

Causality — The Happened-Before Relation

Causality is the fundamental concept linking events in a distributed computation. The happened-before (→) relation (Lamport, 1978) captures which events could causally influence which others.

Causality Rules (Happened-Before →)
Rule 1 (Process Order): If a and b are in same process and a precedes b, then a → b
Rule 2 (Message): If a = send(m) and b = receive(m), then a → b
Rule 3 (Transitivity): If a → b and b → c, then a → c
CONCURRENT: a || b iff NOT(a → b) AND NOT(b → a)
Concurrent means neither could have influenced the other.
Lamport Clock Theorem: a → b ⟹ C(a) < C(b). Contrapositive: C(a) ≥ C(b) ⟹ NOT(a → b).

Distributed Snapshots — Recording Global State

A global snapshot records the state of all processes AND all communication channels at a "consistent" point in time — consistent meaning no message appears to be received without being sent.

Imagine 4 ATM machines in different cities all connected to one bank. You want to take a "photo" of the bank's total money at one moment. But you can't freeze all ATMs simultaneously. Some ATMs are sending/receiving transaction messages. The Chandy-Lamport algorithm takes a consistent snapshot — like a detective reconstructing the exact state at one moment from evidence collected over time, without stopping anything.
Chandy-Lamport Algorithm — Complete Steps
STEP 1 — INITIATOR Pi starts the snapshot:
a) Record own local state: LSi = current state of Pi
b) Send MARKER message on ALL outgoing channels
c) Begin recording messages received on ALL incoming channels
STEP 2 — Process Pj receives MARKER on channel Cij:
CASE A (First MARKER received by Pj):
· Record local state: LSj = current state of Pj
· Set channel state CS(Cij) = EMPTY (no messages "in transit")
· Send MARKER on ALL of Pj's outgoing channels
· Begin recording on all OTHER incoming channels of Pj
CASE B (Pj already recorded state):
· CS(Cij) = all messages received since Pj recorded state until this MARKER
STEP 3 — Termination:
Pj stops recording on Cij after receiving MARKER on it
Algorithm complete when every process received MARKER on all incoming channels
GLOBAL SNAPSHOT = ∪ LSi (all local states) + ∪ CS(Cij) (all channel states)
Key insight: MARKER is a "time fence" — everything before marker is in snapshot, everything after is not.

Applications of Distributed Snapshots

  • Deadlock Detection: Capture global state, check if wait-for graph has a cycle.
  • Termination Detection: Capture global state, check if all processes are idle and no messages in transit.
  • Checkpointing: Save consistent global state so system can recover from failures by rolling back to snapshot.
  • Garbage Collection: Detect distributed objects with no remaining references.
  • Debugging: Post-mortem analysis of distributed system state.

Fault → Error → Failure Chain

Three related but distinct concepts in distributed system reliability:

The FEF Chain (Fault → Error → Failure)
FAULT: The underlying physical or design cause

Example: Buffer overflow vulnerability (software design fault)

Example: The bit flip causes a wrong value to be stored in a variable

Example: Wrong output produced. System crash. Wrong data stored.
Memory: Fault CAUSES Error, Error LEADS TO Failure. Not every fault causes an error (may be dormant). Not every error causes failure (may be masked).

Types of Failures by Behavior (Failure Models)

From simplest (easiest to handle) to most complex (hardest):

Failure Taxonomy — Easiest to Hardest
1. CRASH FAILURE (Fail-Stop):
Server halts and remains halted. No more messages from it.
Example: OS kernel panic, power outage.
Handling: Timeout — if no response after T seconds, assume crashed.
2. OMISSION FAILURE:
Server fails to respond to some requests (but not all).
Send Omission: fails to send some messages
Receive Omission: fails to receive some messages (buffer overflow)
Example: Network packet loss, buffer overflow causing drop.
3. TIMING FAILURE:
Response falls outside specified time interval (too slow or too fast).

Only in SYNCHRONOUS distributed systems.
4. RESPONSE FAILURE:
Response is incorrect. Two subtypes:
Value Failure: Wrong value returned
State Transition Failure: Server deviates from correct state machine
5. BYZANTINE (Arbitrary) FAILURE — HARDEST:
Any behavior — including sending conflicting values to different clients,
colluding with other faulty nodes, actively trying to subvert the system.
Example: Compromised server in a banking system sending wrong balances.
Failure containment: If a component has crash failure, it's easy to contain. Byzantine failures can spread corruption through the entire system.

Fault Tolerance Techniques

  • Redundancy: Replicate components — if one fails, others continue. Hardware redundancy (RAID), Software redundancy (multiple servers), Information redundancy (error-correcting codes).
  • Checkpointing: Periodically save state. On failure, roll back to last checkpoint + replay messages.
  • Message Logging: Log all received messages. On recovery, replay logged messages to reconstruct state.
  • Retry: On timeout, resend request. Requires idempotent operations (safe to repeat).
  • Fail-Stop Design: Design components to halt cleanly on fault detection rather than producing wrong results.

Distributed Mutual Exclusion (DME)

🔮
2026 Prediction — VERY HIGH (Part A Q8): "Explain distributed mutual exclusion" appeared in 2023 Part A Q8. Absent from 2024 and 2025 Part A — the 2-year gap pattern that RTU's 2026 cross-paper analysis confirms as a strong return signal. VERY HIGH confidence for 2026 Part A.

What is Distributed Mutual Exclusion?

Distributed Mutual Exclusion (DME) ensures that at most one process at a time can access a shared critical section (CS) in a distributed system — where there is NO shared memory and NO common clock between processes.

A critical section is a shared resource — like one photocopier shared by 10 offices. Only ONE office can use the photocopier at a time. In a single building, you'd use a lock on the door. In a DISTRIBUTED building (offices in different cities), there's no physical lock — you must use messages to coordinate. That's DME. The challenge: how do you ensure only one "office" (process) uses the "photocopier" (critical section) at a time, using only message-passing?

Requirements of DME Algorithms

DME Requirements
1. SAFETY (Mutual Exclusion): At most ONE process in CS at any time
2. LIVENESS (Progress): Every request to enter CS is eventually granted
3. ORDERING (optional): If request(A) happened before request(B), then A enters CS before B
(FIFO ordering — not all algorithms provide this)

Algorithm 1: Centralized Algorithm

One process is designated as the coordinator. All others ask the coordinator for permission to enter CS.

Centralized Algorithm Steps
To ENTER critical section:
1. Process P sends REQUEST to Coordinator
2. If CS is free: Coordinator sends REPLY(grant) immediately
If CS is busy: Coordinator queues P's request
3. P enters CS after receiving REPLY
To EXIT critical section:
4. P sends RELEASE to Coordinator
5. Coordinator dequeues next waiting process, sends REPLY to it
Messages per CS entry: 3 (REQUEST + REPLY + RELEASE)
+ Easy to implement, fair (FIFO queue). − Single point of failure (coordinator crash). Single bottleneck.

Algorithm 2: Ricart-Agrawala Algorithm (1981)

Fully distributed — no coordinator. Process broadcasts REQUEST to ALL others and waits for REPLY from ALL.

Ricart-Agrawala Algorithm
To ENTER critical section:
1. P timestamps request: T = current Lamport clock value
2. P sends REQUEST(T, P) to ALL other processes
3. P waits to receive REPLY from ALL n-1 other processes
4. P enters CS after all REPLYs received
On receiving REQUEST(T, Q) at process P:
Case A: P is NOT interested in CS → send REPLY immediately
Case B: P is in CS → defer REPLY (send after exiting)
Case C: P wants CS too (competing) →
if T_P < T_Q (P has earlier timestamp): defer REPLY to Q
if T_P > T_Q (Q has earlier timestamp): send REPLY immediately
if tie: lower process ID wins
To EXIT critical section:
5. Send REPLY to all deferred requests
Messages per CS entry: 2(n-1) = n-1 requests + n-1 replies
+ No single point of failure. Provides total ordering. − 2(n-1) messages per CS. Any crash → all blocked.

Algorithm 3: Token Ring Algorithm

Processes are arranged in a logical ring. A single TOKEN circulates around the ring. Only the process holding the token may enter the CS.

Token Ring Algorithm
Setup: Processes P0, P1, P2, ... Pn-1 form a logical ring
Token starts at P0 and circulates: P0→P1→P2→...→P(n-1)→P0→...
To ENTER critical section:
1. Process Pi waits until it receives the TOKEN
2. If Pi wants CS: enter CS, use it, exit, then pass token to P(i+1)
3. If Pi does NOT want CS: immediately pass token to P(i+1)
Messages per CS entry: 1 to n (wait for token to come around)
Bandwidth: 1 message per CS entry (if token circulates efficiently)
+ No starvation (token always comes around). + Simple, efficient when processes often want CS. − Token loss if a process crashes. − Variable wait time (up to n-1 token passes).

Election Algorithms

What is an Election?

Many distributed algorithms require a designated coordinator (leader/coordinator process). An election algorithm selects one process as coordinator when no coordinator exists (initial startup) or when the coordinator crashes.

Imagine a group of employees (processes) who need a team leader (coordinator). Their current leader (old coordinator) has suddenly resigned (crashed). They need to quickly hold an election and pick a new leader — without anyone from outside choosing for them, using only messaging between themselves. The elected leader is typically the one with the highest "employee ID" (process ID).

Key Assumptions

  • Each process has a unique process ID (PID).
  • The process with the highest PID becomes the new coordinator.
  • All processes know all other PIDs (but not which are currently alive).

Algorithm 1: Bully Algorithm (Garcia-Molina 1982)

The process with the highest PID always wins — it "bullies" smaller processes into submission. When a process detects the coordinator has crashed, it starts an election.

Bully Algorithm — 3 Types of Messages
ELECTION: "I want to be coordinator — any process with higher ID?"
OK (Answer): "Yes, I have higher ID — I'll take over"
COORDINATOR: "I am the new coordinator — I won"
Bully Algorithm Steps
Process P detects coordinator crash:
1. P sends ELECTION to all processes with higher PID
2. If no response in T seconds: P declares itself coordinator
→ P sends COORDINATOR to all processes
3. If P receives OK from some Q (Q has higher PID):
→ P waits for COORDINATOR message
→ If no COORDINATOR in T seconds, start election again
Process Q receives ELECTION from P (where P has lower PID than Q):
4. Q sends OK to P (silences P)
5. Q starts its own election (sends ELECTION to all higher than Q)
6. If Q gets no response: Q is highest alive → sends COORDINATOR to all
Messages: O(n²) in worst case. Terminates in O(n) time. Called "bully" because larger processes bully smaller ones into giving up.
Fig: Bully Algorithm — P5 crashes. P3 starts election. P6 (highest alive) eventually wins and sends COORDINATOR to all.

Algorithm 2: Ring Algorithm

Processes are arranged in a logical ring. Election messages circulate around the ring until the winner is identified.

Ring Election Algorithm Steps
Setup: Processes in logical ring P0 → P1 → P2 → ... → P(n-1) → P0
Step 1: Initiator Pi sends ELECTION(Pi) to its neighbor P(i+1)
Step 2: Each process Pj receiving ELECTION(Pk):
If Pk > Pj's PID: forward ELECTION(Pk) to P(j+1)
If Pk < Pj's PID: forward ELECTION(Pj) to P(j+1) (replace with own ID)
If Pk = Pj's PID: Pj is the coordinator! (message went all the way around)
Step 3: New coordinator sends COORDINATOR(Pk) around ring
Messages: O(n) to elect + O(n) to notify = O(2n)
Ring algorithm: simpler, fewer messages, but SLOWER (must traverse full ring). Bully algorithm: faster but more messages.

Distributed Deadlock Handling

🔮
2026 Prediction — CERTAIN (Part A + Part B): Distributed deadlock: 2023 Part C Q3 (full strategies), 2024 Part A Q9 + Part B Q5, 2025 Part B Q4. THREE consecutive years in various forms. THE most consistently tested Unit 5 topic alongside DSM. CERTAIN for 2026.

What is Distributed Deadlock?

Distributed Deadlock occurs when a set of processes in a distributed system are permanently blocked — each process is waiting for a resource held by another process in a cycle, and NO process can ever proceed.

Imagine 4 people standing in a circle, each holding one chopstick and waiting for the chopstick held by the person to their right. Nobody moves. Nobody can eat. Nobody can release their chopstick because they're all frozen waiting. THAT's deadlock — a circular wait with no way out. In DS, processes wait for messages/resources held by other processes on different machines, creating a distributed deadlock cycle that's much harder to see because no single machine has the full picture.

Coffman's Necessary Conditions (All 4 must hold)

Coffman Conditions — Memorize MHNC
1. MUTUAL EXCLUSION: Resources cannot be shared — only one process at a time
2. HOLD AND WAIT: Process holds resources while waiting for more
3. NO PREEMPTION: Resources cannot be forcibly taken away
4. CIRCULAR WAIT: Cycle of processes: P1 waits for P2, P2 waits for P3, ... Pn waits for P1
All four MUST hold simultaneously for deadlock. Remove any one → deadlock impossible.

Strategy 1: Distributed Deadlock Prevention

Ensure at least one of Coffman's conditions NEVER holds:

  • Eliminate Circular Wait: Impose a global ordering on resources (e.g., numbered 1 to N). Processes must request resources in increasing order. No cycle possible.
  • Eliminate Hold-and-Wait: Process must request ALL resources it will ever need at once before starting. Impractical in DS (can't predict future needs).
  • Allow Preemption: If a process with high priority needs a resource, forcibly take it from a lower-priority process. The preempted process must restart. Possible only for resources that allow rollback (e.g., CPU time, not printers).

Strategy 2: Deadlock Avoidance (Wait-Die / Wound-Wait)

Use process timestamps to prevent circular wait in DS. Every process gets a unique timestamp at creation time (Lamport clock value). Older processes have smaller timestamps.

Wait-Die (Non-Preemptive)
Process Pi requests resource held by Pj:
If Ti < Tj (Pi is OLDER): Pi WAITS (older waits for younger)
If Ti > Tj (Pi is YOUNGER): Pi DIES (aborts and restarts with same timestamp)
Mnemonic: "Old wait, young die"
Only older processes wait — younger ones die. Eventually the oldest process always gets resource. No deadlock possible because no circular wait (only older→younger direction).
Wound-Wait (Preemptive)
Process Pi requests resource held by Pj:
If Ti < Tj (Pi is OLDER): Pi WOUNDS Pj (preempts — aborts Pj, gets resource)
If Ti > Tj (Pi is YOUNGER): Pi WAITS for Pj to release
Mnemonic: "Old wound young, young wait for old"
Older processes preempt younger ones. Preempted process restarts. No deadlock because younger processes never cause older ones to wait indefinitely.

Wait-DieOlder WAITS (non-preemptive)Younger DIES (aborts)Non-preemptive
Wound-WaitOlder WOUNDS (preempts younger)Younger WAITSPreemptive

Strategy 3: Distributed Deadlock Detection

Allow deadlocks to occur but detect them using the Wait-For Graph (WFG). In a distributed WFG, edges represent "process waiting for resource held by another process." A cycle in the WFG means deadlock.

Challenge: No single node has the complete global WFG — each node only sees its local part. We need a distributed algorithm to collect and detect cycles in the global WFG.

Chandy-Misra-Haas Algorithm for Deadlock Detection
Terminology: Pi is waiting for Pj (Pi → Pj in WFG)
PROBE Message: PROBE(i, j, k) — sent to detect deadlock
i = process that might be deadlocked (initiator)
j = process that sent this probe
k = process receiving this probe
Algorithm:
Step 1: Pi detects it is waiting for Pj
Pi sends PROBE(i, i, j) to Pj
Step 2: Pj receives PROBE(i, j, k=Pj):
If Pj is waiting for Pl (for all l that Pj is waiting for):
Send PROBE(i, Pj, Pl) to Pl
[Probe propagates along wait-for edges]
Step 3: DEADLOCK DETECTED:
If Pi receives PROBE(i, j, i) for some j
(Probe returned to initiator!) → CYCLE EXISTS → Pi is deadlocked
When probe returns to initiator, a cycle has been found in the global WFG. Algorithm finds ALL deadlocks, not just cycles involving the initiator.

Deadlock Recovery

Once deadlock is detected, the system must recover by breaking the cycle:

  • Victim Selection: Choose one or more processes to abort. Selection criteria: youngest process (lowest cost to restart), process with least work done, process requesting the most resources.
  • Process Rollback: Roll back victim to a previously saved checkpoint and restart.
  • Resource Preemption: Take resources from victim process and give to others. Victim may need to restart.

Distributed Termination Detection

The Termination Problem

Distributed Termination Detection determines when a distributed computation has completed — when all processes are idle AND all communication channels are empty (no messages in transit).

Imagine 10 workers in different cities collaborating on a project. How do you know when they're ALL done? It's not enough that each individual reports "I'm done" — because messages might still be travelling between them (a worker sent a task to another worker, and that task is in the mail). You need to know: everyone is idle AND no work is "in flight" anymore. That's the termination detection problem.

Why is it Hard?

Termination seems simple — just wait for all processes to signal "done." But this fails because:

  • Process A might be idle, but a message sent by Process B (also idle) is still in transit to A. When A receives the message, A becomes active again — so A was not truly done.
  • No global state is visible to any single process.
Termination Condition
Computation has TERMINATED iff:
Condition 1: ALL processes are in PASSIVE (idle) state
AND
Condition 2: ALL communication channels are EMPTY (no messages in transit)
Process States:
ACTIVE: process is computing or has unprocessed messages
PASSIVE: process is idle, waiting for messages
A passive process becomes active when it receives a message. An active process becomes passive when it finishes its current computation.

Dijkstra-Scholten Algorithm

Uses a spanning tree of active processes. Root is the initiator. Each process tracks its "deficit" (tokens sent minus tokens received).

Dijkstra-Scholten Algorithm — Key Ideas
Environment sends initial message to Root (activates Root)
Root "owns" all processes it activates (forms spanning tree)
Deficit(P) = (messages sent by P) - (acknowledgements received by P)
TERMINATION DETECTED when:
Root becomes passive AND Deficit(Root) = 0
(Root sent no unacknowledged messages AND root is idle)
Process P becomes leaf in tree and sends acknowledgement to parent when:
P is passive AND P has no unacknowledged messages (Deficit(P)=0)
Tree shrinks as processes become idle. When tree reduces to root and root becomes idle → termination.

Token-Based Termination: Safra's Algorithm

Processes are arranged in a ring. A token circulates. Color system (black/white) tracks whether a process has "become active again" after the token passed it.

Safra's Algorithm
Token has: (color, count)
Process state: white (clean) or black (sent message to earlier process)
P0 initiates: send token(WHITE, 0) to P1
Each process Pk receives token:
If Pk is active: wait until passive, then forward
new_count = token.count + (messages sent - messages received) by Pk
new_color = BLACK if Pk is black; else token color
Forward token(new_color, new_count) to P(k+1)
P0 detects termination when:
Token returns AND token.color = WHITE AND token.count = 0
AND P0 is white AND P0 is passive
count=0 means messages sent = messages received globally. WHITE = no process "went black" during this round.
🔗
Connection: Termination Detection ↔ Distributed Snapshots

Distributed Snapshots (Chandy-Lamport) can also detect termination: take a snapshot, check if all processes are passive AND all channel states are empty. If yes → terminated! In practice, dedicated termination algorithms (Dijkstra-Scholten, Safra) are more efficient because they don't need to record full system state.

Prevention: Impose global ordering on resources → eliminates circular wait. Or request all resources at once → eliminates hold-and-wait.

Avoidance (Wait-Die / Wound-Wait):

Wait-Die: Older waits for younger; younger DIES (aborts+restarts)
Wound-Wait: Older WOUNDS (preempts) younger; younger waits for older
Both prevent circular wait. Only unidirectional waiting direction possible.

Detection (Chandy-Misra-Haas): PROBE(i,j,k) messages follow wait-for edges. If probe returns to initiator → cycle → deadlock. Recovery: Abort youngest victim in cycle.

Memory Consistency Model defines what values a read can return given a history of writes in a shared/distributed memory system.

Strict: Most recent write returned — impossible in DS (no global clock)
Sequential (Lamport): All see same global op order — standard, Java/C++
Causal: Causally related writes in order; concurrent writes can differ
Release: Sync only at lock acquire/release — efficient, TreadMarks
Eventual: All replicas converge if no new writes — Cassandra, Dynamo
Trade-off: Stronger = easier to program, lower performance. Weaker = harder, better performance.
