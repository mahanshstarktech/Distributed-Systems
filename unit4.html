A System Performance Model

Why Do We Need a Performance Model?

Before deciding how to schedule processes across distributed nodes, we need a way to measure and predict system performance. A system performance model provides mathematical tools to analyze process execution time, communication costs, and overall system efficiency.

Imagine you're a delivery manager deciding which delivery person should handle which package. Before assigning, you need to know: How fast can each person travel? How long does handing off a package between drivers take? What's the total time for a route? The performance model is exactly this — a mathematical framework that tells us the "cost" of running a process on each machine and the "cost" of communication between machines.

Key Performance Metrics

Performance Model Parameters
Execution Time (E): Time to execute a process on a given processor
Communication Time (C): Time to send data between two processors
Latency (L): Fixed overhead to initiate a message (setup cost)
Bandwidth (B): Data transfer rate (bytes/second)
Message Transfer Time: T = L + (message_size / B)
Total Time for Parallel Program:
T_total = T_computation + T_communication + T_synchronization
Speedup (S): S = T_serial / T_parallel
Efficiency (E): E = S / P (P = number of processors)
Ideal speedup: S = P (efficiency = 1.0 = 100%)
Amdahl's Law: Max speedup limited by serial portion. S ≤ 1/(f + (1-f)/P) where f = fraction that must be serial

Task Graph Model

A distributed program can be modelled as a task graph (DAG) where nodes represent tasks and edges represent data dependencies between tasks. Scheduling algorithms use this graph to optimally assign tasks to processors.

Fig: Task Dependency Graph. e = execution time, c = communication cost on edges. Scheduler assigns tasks (T1–T4) to processors minimizing total completion time.
🔗
Connection: Performance Model → Scheduling Decisions

The performance model is the INPUT to scheduling algorithms. Static scheduling uses the task graph's execution times and communication costs (known at compile time). Dynamic scheduling monitors actual runtime loads. Without the performance model, you can't decide where to run what. Message transfer time formula (T = L + size/B) is also the basis for understanding why minimizing inter-node communication is critical in static scheduling.

Static Process Scheduling with Communication

🔮

What is Static Process Scheduling?

Static process scheduling (also called compile-time or offline scheduling) involves deciding at compile time (before execution) which process runs on which processor. The task assignment is fixed — no migration occurs at runtime.

Imagine planning a wedding banquet. You assign table numbers to guests BEFORE the event — guest A sits at table 3, guest B at table 1, etc. This is static scheduling — everything decided upfront. You can plan it perfectly if you know exactly how many guests there are and their preferences. But if a guest cancels or an extra guest shows up, you have a problem. Static scheduling is perfect for predictable workloads but fails with unpredictable ones.

Characteristics of Static Scheduling

  • Information known at compile time: Number of processes, their execution time, communication requirements, and dependencies.
  • No runtime overhead: No monitoring of system load, no migration during execution.
  • Optimal for regular workloads: Works well when workload is predictable (scientific simulations, matrix operations).
  • Fails with irregular workloads: Cannot adapt if some tasks take longer than expected or if hardware fails.

Scheduling Goals

Static Scheduling Objectives
1. Minimize total execution time (makespan) of the program
2. Maximize processor utilization (minimize idle time)
3. Minimize inter-processor communication (place communicating tasks on same/nearby processors)
4. Satisfy all precedence constraints (respect task dependencies — edges in DAG)
KEY TENSION: Parallelism ↔ Communication
Too much parallelism → excessive communication → worse performance
Too little parallelism → underutilized processors
NP-complete in general — no polynomial-time optimal algorithm exists. Heuristics used in practice.

Static Scheduling Algorithms

Common Static Scheduling Approaches
1. LIST SCHEDULING (most common heuristic):
Assign priority to each task (e.g., by critical path length)
Always schedule the highest-priority ready task on the first free processor
"Ready" = all predecessor tasks have completed
2. CLUSTER SCHEDULING:
Group communicating tasks into clusters
Map clusters to processors to minimize inter-cluster communication
3. CRITICAL PATH SCHEDULING:
Critical path = longest path in the task DAG (determines minimum completion time)
Schedule tasks on critical path first (they have no slack)
4. MAPPING (DAG Partitioning):
Partition task graph into P parts (one per processor)
Minimize edges between partitions (= minimize communication)
Example: METIS graph partitioning tool used in scientific computing

Static vs Dynamic Scheduling — Comparison

AspectStatic SchedulingDynamic Scheduling
When decidedCompile time (before execution)Runtime (during execution)
Information usedKnown task graph, static estimatesActual runtime load information
MigrationNone — fixed assignmentProcesses can migrate between nodes
OverheadLow (one-time compile cost)Higher (runtime monitoring + migration)
AdaptabilityNone — fails if estimates wrongAdapts to actual workload
OptimalityNear-optimal if model accurateGood for irregular workloads

🔮

What is Dynamic Load Sharing?

Dynamic load sharing redistributes tasks between processors at runtime based on current system state — when some nodes are overloaded and others are idle, tasks migrate to balance the load. Unlike static scheduling, it responds to actual workload conditions rather than compile-time estimates.

Imagine a restaurant with 5 waiters. At lunch, tables near the entrance get overwhelmed while tables at the back are empty. Dynamic load balancing is like the manager who watches what's happening and says: "Waiter A, you're too busy — Waiter B will take your next two tables." This "boss" is the load balancer who constantly monitors and redistributes work so everyone stays busy but not overwhelmed.

Key Components of Dynamic Load Sharing

4 Policy Components
1. TRANSFER POLICY: When to transfer a process?
Threshold-based: Transfer when load > upper threshold OR node has idle capacity
Threshold value is a key design parameter
2. SELECTION POLICY: Which process to transfer?
Typically: Youngest process, most recently created, or next in queue
Avoid transferring processes that have accumulated significant state (expensive migration)
3. LOCATION POLICY: Where to transfer?
Find a node below lower threshold (has spare capacity)
Poll random nodes OR use centralized load information service
4. INFORMATION POLICY: How to gather load information?
On demand: Request load info when needed (sender or receiver polls)
Periodic broadcast: All nodes broadcast load every T seconds
Event-driven: Broadcast only when state changes significantly
RTU exam: "What is threshold?" (2024 Q6) — Threshold is the load level that triggers load sharing actions. Upper threshold = overloaded. Lower threshold = underloaded.

Sender-Initiated Algorithm (RTU 2023 Part C Q5)

The overloaded node (sender) initiates the process transfer. When a node detects it is overloaded (load exceeds upper threshold), it proactively searches for a lightly loaded node to send work to.

Fig: Sender-Initiated Algorithm — Overloaded Node A finds and sends work to lightly loaded Node B
Sender-Initiated Algorithm — Detailed Steps
Step 1: Node detects its load exceeds UPPER threshold
Step 2: Node randomly selects another node and sends PROBE message ("are you lightly loaded?")
Step 3: If probed node is below LOWER threshold → it replies OK
Step 4: Sender migrates a process to the responder
Step 5: If no suitable node found after N probes → keep process locally
Advantages:
+ Very effective at distributing load away from overloaded nodes
+ Simple to implement
Disadvantages:
- When ALL nodes are overloaded, generates a probe message storm (flooding)
- Wastes network bandwidth at exactly the worst possible time

Receiver-Initiated Algorithm (RTU 2023 Part C Q5)

The lightly loaded node (receiver) initiates the transfer. When a node becomes idle or underloaded (load below lower threshold), it actively seeks work from overloaded nodes.

Receiver-Initiated Algorithm — Detailed Steps
Step 1: Node detects its load falls below LOWER threshold (idle/underloaded)
Step 2: Node randomly selects another node and sends PROBE message ("do you have excess work?")
Step 3: If probed node is above UPPER threshold → it replies with a process to transfer
Step 4: Receiver accepts and executes the migrated process
Step 5: If no overloaded node found after N probes → receiver waits and tries later
Advantages:
+ Effective when system is lightly loaded — idle nodes actively seek work
+ No flood during heavy load (overloaded nodes don't initiate probes)
Disadvantages:
- When ALL nodes are lightly loaded, many idle nodes simultaneously send probes → flood
- Less effective under high overall load

Symmetric Algorithm — Best of Both Worlds

The symmetric algorithm combines both sender-initiated and receiver-initiated approaches:

  • If load > upper threshold → act as sender (push work out)
  • If load < lower threshold → act as receiver (pull work in)
  • If lower ≤ load ≤ upper → satisfied node (neither pushes nor pulls)
Three-Zone Load Model (Symmetric Algorithm)
Load: 0 ─────────────[Lower]──────────────[Upper]──────────── Max
Zone: ←── Receiver Zone ──→←── Satisfied Zone ──→←── Sender Zone ──→
Action: Pull work in No action Push work out
Most practical load sharing systems use the symmetric approach. Real example: Condor HTC system.
AlgorithmWho InitiatesBest WhenWorst When
Sender-InitiatedOverloaded nodeSystem heavily loadedALL nodes overloaded (probe storm)
Receiver-InitiatedUnderloaded nodeSystem lightly loadedALL nodes idle (probe storm)
SymmetricBoth — threshold-triggeredVariable/mixed loadOverhead at both thresholds simultaneously

In dynamic load sharing, a threshold is a load level that triggers a load-balancing action:

Upper threshold: Maximum acceptable load for a node. If current load exceeds this, the node is considered "overloaded" — in sender-initiated algorithm, it starts probing other nodes to transfer work.

Lower threshold: Minimum desired load. If current load falls below this, the node is "underloaded" or idle — in receiver-initiated algorithm, it actively seeks work from other nodes.

Distributed Process Implementation

🔮
2026 Prediction — HIGH (Part B): Distributed process implementation appeared in 2024 Part C Q3 and 2025 Part B Q5. Two consecutive years with different depth. Based on RTU's pattern of alternating weight (Part C → Part B), HIGH confidence for Part B Q3 in 2026.

What is Distributed Process Implementation?

Distributed process implementation covers how processes are created, managed, communicated with, and migrated across multiple machines in a distributed system. It includes the mechanisms that the distributed OS or middleware provides to support running programs on a collection of networked nodes.

When you run a program on your laptop, the OS handles everything — creating a process, allocating memory, scheduling CPU time. In a distributed system, a process might start on one machine, use files on a second machine, communicate with a service on a third, and if the first machine gets overloaded, the OS might move the process to a fourth machine. "Distributed process implementation" is how all this is made to work seamlessly.

Process Creation in Distributed Systems

Process Creation Mechanisms
1. LOCAL FORK: Create new process on same node as parent (traditional Unix fork)
Simple; parent and child share initial address space context
2. REMOTE PROCESS CREATION: Create process on specified remote node
rexec (Remote Execute): explicitly name the target host
Steps: Check target availability → Transfer executable → Create process → Return PID
3. ADAPTIVE PROCESS CREATION:
System automatically selects best node based on current load
Client asks scheduler: "Create process on best available node"
Foundation of dynamic load balancing

Process Migration

Process migration is the transfer of a running (or suspended) process from one node to another. It is the key mechanism enabling dynamic load sharing.

Process Migration — Types and Steps
WEAK MIGRATION (Suspend and Restart):
1. Suspend process at source node
2. Checkpoint: Save process state (code, data, stack, registers, file descriptors)
3. Transfer checkpoint to destination node
4. Restart process at destination (process loses in-flight messages)
Simpler but causes pause. Used when process can be restarted cleanly.
STRONG MIGRATION (Live/Transparent Migration):
1. Capture entire process state while it runs (like VM live migration)
2. Transfer state and open file/socket connections to destination
3. Resume process at destination — appears to continue uninterrupted
More complex; requires OS support for connection forwarding
Cost of migration: Transfer of code + data + stack. Must transfer entire address space if no shared virtual memory.

Issues in Process Migration

  • Address Space Transfer: The entire virtual address space must be transferred — can be large (gigabytes). Optimization: copy-on-demand (transfer only pages actually accessed).
  • Open File Handling: Process may have open files on the source node. After migration, file access must be forwarded back to source or files replicated.
  • Network Connections (Sockets): TCP connections have source IP — after migration, connections appear to come from wrong IP. Solution: connection forwarding or mobile IP.
  • Message Forwarding: Messages sent to migrated process must be forwarded to new location. Home node keeps forwarding record.
  • Performance Cost: Migration itself takes time and uses bandwidth. Must only migrate when benefit (load reduction) exceeds migration cost.
🔗
Connection: Process Migration → VM Live Migration (Unit 4 CC) → Transparency (Unit 2 DS)

Process migration in DS is the inspiration for VM live migration in cloud computing (vMotion). Both aim for migration transparency — users/processes never notice the move. The difference: VM migration moves an entire virtual machine (OS + processes); process migration moves a single process. Both achieve the same goal: load balancing without disrupting service, which directly implements migration transparency from Unit 2.

Distributed File System — Introduction

🔮
2026 Prediction — CERTAIN (Part A + Part C): DFS appeared in 2023 Part A Q4, 2024 Part A Q5, and 2025 Part C Q3 — three consecutive years in some form. It is the most consistent Unit 4 topic. CERTAIN to appear in 2026.

What is a Distributed File System (DFS)?

A Distributed File System (DFS) allows users and programs on client machines to access files stored on one or more remote servers, making those remote files appear as if they were stored locally. The underlying network activity is completely transparent to users and applications.

Imagine you're editing a Word document that appears to be on your computer. But actually, it's stored on a server in another building — and 50 of your colleagues can also access and edit it. When you save, it goes to the server. When someone else makes a change, you see it reflected. You never notice the file isn't local. That's a distributed file system. Google Drive, Dropbox, and OneDrive are DFS examples you use every day — but enterprise DFS is far more complex.

DFS vs Centralized File System (2023 Part A Q4)

AspectCentralized File SystemDistributed File System
LocationFiles on one machineFiles on multiple servers

ScalabilityLimited to one machine's storageAdd more servers for more capacity

DFS Transparencies

A well-designed DFS should provide multiple forms of transparency (from Unit 2):

DFS Transparency Types
1. ACCESS Transparency: Same operations (open/read/write/close) for local AND remote files
2. LOCATION Transparency: File accessed by name — user doesn't know which server holds it
3. MIGRATION Transparency: File can move between servers without changing its name/path
4. REPLICATION Transparency: Multiple copies exist; user sees one consistent file
5. FAILURE Transparency: File service continues despite server failures (via replication)
6. CONCURRENCY Transparency: Multiple clients access/modify file; each sees consistent view
7. PERFORMANCE Transparency: System can replicate/cache to improve performance transparently
8. SCALING Transparency: More servers added without user-visible changes

Characteristics of a Good DFS

  • High availability: Files accessible even if some servers are down (replication)
  • Reliability: Data not lost despite crashes (redundant storage, journaling)
  • High performance: Client-side caching reduces server load and network traffic
  • Scalability: Handle millions of files and thousands of clients
  • Consistency: Multiple clients see consistent view of files
  • Security: Access control — only authorized users access files

DFS Design and Implementation

Architecture of a DFS

A typical DFS has a client-server architecture. Clients access files via the DFS client software; file servers store the actual data. Communication happens via RPC (Remote Procedure Calls).

Fig: DFS Architecture — VFS layer routes file operations; DFS client communicates with file server via RPC; cache reduces network traffic

DFS Naming and Namespace

Naming Approaches in DFS
1. LOCATION-DEPENDENT NAMING (Not transparent):
File name includes server: //server1/files/document.txt
User must know server — breaks location transparency
If file moves to server2, name changes → breaks existing references
2. LOCATION-INDEPENDENT NAMING (Transparent):
File accessed by abstract name: /home/student/document.txt
Namespace service maps name → server location
File can move servers without changing name
NFS: /mnt/files/document.txt (mount point hides server)
AFS: /afs/cs.cmu.edu/user/student/document.txt (global namespace)
3. GLOBAL NAMESPACE:
Single uniform namespace visible to all clients worldwide
AFS's /afs/... tree spans multiple institutions
Best transparency: global namespace. Most common: mount-point based (NFS style).

Client-Side Caching — The Key to Performance

The most critical optimization in DFS is client-side caching. Instead of going to the file server for every read, the client caches recently accessed file blocks locally. This dramatically reduces network traffic and server load.

Cache Consistency Problem and Solutions
PROBLEM: Client A caches block 5 of File X at time T.
Client B modifies block 5 at time T+1.
Client A still has old (stale) copy — sees inconsistent data!
SOLUTION 1 — WRITE-THROUGH:
Every write immediately sent to server; cache always consistent with server
Advantage: Always fresh data on server
Disadvantage: No performance gain for writes (network on every write)
SOLUTION 2 — WRITE-BACK (Delayed Write):
Writes cached locally, flushed to server periodically or on close
Advantage: Great write performance
Disadvantage: Crash loses unflushed data; other clients may see stale data
SOLUTION 3 — CALLBACK MECHANISM (AFS):
Server notifies clients when a file they have cached is modified
Client invalidates its cache on callback; fetches fresh copy next access
Advantage: Always fresh, good performance; only invalid when file changes
Disadvantage: Server must track who has cached what (stateful server)
SOLUTION 4 — VALIDATION ON OPEN (NFS):
On file open, check if cached copy still valid (compare timestamps with server)
Disadvantage: Still stale between opens; checking on every access is expensive
Trade-off: Server state vs cache freshness. AFS uses callbacks (stateful). NFS uses timestamp check (stateless).

Transaction Service and Concurrency Control

Transactions in Distributed File Systems

A transaction is a sequence of file operations (reads and writes) that must execute as an atomic unit — either all operations complete successfully, or none of them take effect. This is essential when multiple clients access shared files.

Think of transferring money between bank accounts (files). You need to deduct from account A AND add to account B. If the system crashes after deducting from A but before adding to B, money is lost! A transaction guarantees that either BOTH operations happen or NEITHER does — no partial results. This is ACID.

ACID Properties

ACID Properties (Essential for Exams)
A — ATOMICITY: Transaction is all-or-nothing
Either all operations complete OR none do (rollback on failure)
Implemented via: Undo logs, write-ahead logging (WAL)
C — CONSISTENCY: Transaction takes DB from one valid state to another
Invariants hold before and after every transaction
Example: Total money in system unchanged after bank transfer
I — ISOLATION: Concurrent transactions execute as if in serial order
One transaction's partial results invisible to other transactions
Implemented via: locking, timestamps, optimistic concurrency control
D — DURABILITY: Committed transactions survive system crashes
Committed data written to stable (non-volatile) storage
Implemented via: write-ahead logging, checkpointing

Concurrency Control Techniques

Three Main Concurrency Control Approaches
1. TWO-PHASE LOCKING (2PL):
Phase 1 (Growing): Acquire locks, never release
Phase 2 (Shrinking): Release locks, never acquire new ones
Read locks (shared) + Write locks (exclusive)
Guarantees serializability but can cause deadlock
2. TIMESTAMP ORDERING:
Each transaction assigned a unique timestamp at start
Conflicting operations must execute in timestamp order
If conflict detected → abort younger transaction; restart later
Advantage: No deadlock. Disadvantage: High abort rate under contention
3. OPTIMISTIC CONCURRENCY CONTROL:
Execute without locks; validate at commit time
Phases: Read → Validate (check for conflicts) → Write or Rollback
Great for low-contention workloads; wastes work on rollback if conflict

Two-Phase Commit (2PC) — Distributed Atomicity

When a transaction spans multiple servers, Two-Phase Commit (2PC) ensures all-or-nothing across all participants.

Two-Phase Commit Protocol
PHASE 1 (PREPARE / VOTING):
Coordinator sends PREPARE to all participants
Each participant: checks if it can commit → replies YES or NO
YES means: "I've done the work, logged to stable storage, ready to commit"
PHASE 2 (COMMIT / ABORT):
If ALL voted YES → Coordinator sends COMMIT to all
If ANY voted NO → Coordinator sends ABORT to all
Each participant: commits or aborts accordingly, releases locks
Problem: Coordinator crash between Phase 1 and Phase 2 → participants blocked!
Solution: Three-Phase Commit (3PC) or Paxos-based consensus
2PC is blocking — if coordinator crashes, participants wait indefinitely (hold locks)

Data and File Replication

Why Replicate?

File replication creates multiple copies of files on different servers. This provides fault tolerance (if one server fails, others continue), improved performance (clients read from nearest replica), and scalability (distribute read load across replicas).

Libraries keep multiple copies of popular books so more people can read simultaneously and so if one book is damaged, the library still has copies. File replication does the same for data. If the New York server has 1000 clients and the London server has 1000 clients, keeping a copy in both cities means each group reads locally (fast!) and if New York's server fails, London's copy is still available.

Replication Protocols

Replication Strategies
1. PRIMARY-BACKUP (Primary Copy):
One replica is PRIMARY (handles all writes)
Others are BACKUPS (receive updates from primary)
Simple; no write conflicts; but primary is bottleneck and single point of failure
Used by: MySQL primary-replica, PostgreSQL streaming replication
2. QUORUM-BASED REPLICATION:
Read quorum (r) + Write quorum (w) where r + w > n (n = total replicas)
Read from any r replicas; write to any w replicas
Guarantees: reads always see most recent write
Example: n=5, r=3, w=3: r+w=6>5 ✓; can read and write from any majority
3. EVENTUAL CONSISTENCY:
Replicas may be temporarily inconsistent; will eventually converge
Allow writes anywhere; propagate changes asynchronously
Conflict resolution: last-write-wins, vector clocks, application-specific merge
Used by: Amazon Dynamo, Apache Cassandra, DynamoDB
Trade-off: High availability + performance vs strong consistency

Replication in DFS — Read and Write Operations

StrategyReadWriteConsistencyUse Case
Primary-BackupFrom any backup (fast)To primary only; propagated to backupsStrong (after propagation)DB replication, HDFS

⚡
Exam Trick — CAP Theorem (Often Asked Implicitly)

CAP Theorem: A distributed system can guarantee at most 2 of 3: Consistency, Availability, Partition tolerance. Network partitions are unavoidable → must choose CA or CP or AP. Strong replication = CP (consistent but may be unavailable during partition). Eventual consistency = AP (always available but may be inconsistent). Mention this when answering replication or DFS consistency questions for extra marks.

Sun Network File System (NFS)

🔮
2026 Prediction — HIGH (Part A/B): NFS has been mentioned in Part A questions across all years (as "DFS examples"). NFS vs AFS comparison is a classic DS exam question and could appear as Part B in 2026 given the DFS topic is anchored every year.

What is NFS?

NFS (Network File System) was developed by Sun Microsystems in 1984 and remains the most widely used distributed file system on Unix/Linux systems. Its defining design choice is that the server is stateless — it keeps no information about client state between requests.

NFS is like ordering food at a restaurant where the waiter has absolutely no memory. Each time you call the waiter, you must provide your full order from scratch: "I'm at table 5, I ordered the pasta, I want to add more sauce to it." The waiter processes your request and then instantly forgets who you are. Next time, you must repeat everything. This is stateless — it's less efficient but much simpler to recover from crashes.
Fig: NFS Architecture — Stateless server; client uses VFS and cache; requests are self-contained

NFS Design Decisions

Key NFS Design Choices
1. STATELESS SERVER:
Server keeps NO per-client state between requests
Every request is self-contained: (file handle, offset, count)
Advantage: Simple crash recovery — server just restarts; clients retry
Disadvantage: No callbacks possible; clients can't know when file changes
2. FILE HANDLES:
Server gives client an opaque "file handle" on open
Handle encodes: filesystem ID + inode number + generation number
Client uses handle in all subsequent requests
3. IDEMPOTENT OPERATIONS:
All NFS operations designed to be safe to retry
Read at offset X always returns same result → safe to retry
Avoids duplicate execution problems in UDP-based protocols
4. CLIENT-SIDE CACHING:
Clients cache file blocks locally
Cache validation: On file access, compare timestamp with server
If server's mtime > cached time → invalidate cache, fetch from server
Default: check freshness every 3-30 seconds (close-to-open consistency)
5. CLOSE-TO-OPEN CONSISTENCY:
Guarantee: if client A closes a file and client B opens it,
B always sees A's latest writes
Between opens: may see stale data (acceptable for most uses)

NFS Limitations

  • Scalability: NFS server can handle ~100 clients before becoming bottleneck (stateless doesn't mean unlimited scale)
  • Cache Consistency: Not strict — clients can see stale data for up to 30 seconds
  • No global namespace: Each client mounts shares manually with specific server addresses
  • Security: Original NFS had no authentication (any client could claim to be any user)

Andrew and Coda File Systems

Andrew File System (AFS)

AFS (Andrew File System) was developed at Carnegie Mellon University in the 1980s and designed to scale to 50,000+ clients — far beyond NFS's capabilities. Its key innovation: whole-file client caching with server callbacks.

While NFS is like a stateless waiter who forgets you instantly, AFS is like a personal assistant who remembers your preferences and proactively tells you when something changes. When you open a file in AFS, the server gives you the ENTIRE file (like downloading it locally). You work on your local copy. If ANYONE else changes the file on the server, the server immediately calls you back ("Hey, your cached copy is now stale!"). This is much more efficient than NFS's constant checking.
AFS Key Design Decisions
1. WHOLE-FILE CACHING on client:
On first open: entire file transferred to client's local disk
All reads/writes go to LOCAL disk copy (very fast — no network)
On close: modified file sent back to server (write-on-close semantics)
2. CALLBACK MECHANISM (Stateful Server):
Server tracks which clients have cached which files
If any client modifies file → server immediately sends CALLBACK to all other holders
Client receiving callback: marks its cached copy as INVALID
Next access → client fetches fresh copy from server
Result: Clients ALWAYS see latest version (strong consistency)
3. GLOBAL NAMESPACE:
/afs/institution.edu/user/file — works from ANY AFS client worldwide
Location-independent — file can move between servers, path unchanged
4. VOLUMES (Administrative Units):
Files grouped into volumes (typically per-user or per-project)
Volumes can be moved between servers for load balancing
Volume location stored in global database (VLDB)
5. SCALABILITY: Designed for 50,000+ clients
Callbacks dramatically reduce server load (no polling)
Most accesses served from local disk cache → server rarely bothered

Coda File System

Coda is an extension of AFS that adds support for disconnected operation — a client can continue working with cached files even when disconnected from the network (on a laptop on an airplane), and resynchronizes when reconnected.

Coda Key Features
1. DISCONNECTED OPERATION:
Client caches files before disconnecting (or caches on-demand during hoarding)
All reads/writes while disconnected go to local cache
On reconnection: REINTEGRATION — local changes propagated to server
Conflicts (server changed while disconnected): application-specific resolution
2. HOARDING:
Before going offline, user specifies files to hoard (prefetch)
Hoard daemon fills cache with likely-needed files
3. CML (Client Modify Log):
All modifications during disconnection logged in CML
On reintegration: replay CML at server to apply changes
4. CONFLICT DETECTION:
If server and client both modified same file: CONFLICT
Coda detects conflict, alerts user/application to resolve
5. WEAK CONNECTIVITY:
Works even over slow (bandwidth-limited) links
Defers write propagation when link is slow
Coda was ahead of its time — same concepts are in Dropbox, Google Drive, OneDrive (2020s cloud sync)
Fig: Coda three-state model — Connected (normal), Disconnected (local cache), Reintegrating (sync on reconnect)

Coda RPC2 Side Effects:
RPC2 is Coda's higher-level RPC mechanism built on top of standard RPC. "Side effects" in Coda's RPC2 context refer to the CML (Client Modify Log) entries created as a side effect of every write operation during disconnected mode.

Side effects of RPC2 operations:
1. Client-side: Every write logged in CML — side effect of normal write RPC. Log grows during disconnection.
2. Cache update: Every RPC that reads data updates local cache — side effect is cache modified.
3. Callback state: Open file RPC registers callback — side effect is server now tracking client.

Coda File Identifier (FID):
A Coda FID (File Identifier) is a unique, location-independent identifier for every file/directory in the Coda namespace.

Structure: FID = (Volume ID, Vnode number, Unique number)
· Volume ID: identifies which volume contains the file
· Vnode number: inode-like index within the volume
· Unique number: version counter — incremented when file is deleted and inode reused

Implementation: When client opens a file, it receives its FID. All subsequent operations reference the FID (not filename). FID is stable even if file is renamed or volume is migrated to different server.

Resolution: Client looks up FID → queries Volume Location Database (VLDB) using Volume ID → gets server address → contacts server with Vnode number.

Distributed Process Implementation: Mechanisms for creating, managing, and migrating processes across distributed nodes.

Process creation: Local fork, Remote process creation, Adaptive creation (load-based). Process migration: Weak (suspend+checkpoint+restart) or Strong (live transparent transfer). Issues: address space transfer, open files, TCP connections, message forwarding.

Dynamic Load Sharing redistributes tasks at runtime based on actual system load. Uses thresholds to trigger transfer actions.

Sender-Initiated Algorithm: Overloaded node (load > upper threshold) proactively finds and sends work to a lightly loaded node.

Steps: Detect overload → Poll random nodes → Find light node → Transfer process
✓ Good under heavy load ✗ Probe flood when ALL nodes overloaded

[Draw: Node A (OVERLOADED) → PROBE → Node B → OK → Process Migration]

Receiver-Initiated Algorithm: Idle node (load < lower threshold) actively requests work from overloaded nodes.

Steps: Detect underload → Poll random nodes → Find overloaded node → Request work
✓ Good under light load ✗ Probe flood when ALL nodes idle

[Draw: Node B (IDLE) → PROBE → Node A → transfers process → Node B executes]

Symmetric Algorithm: Combines both — above upper threshold acts as sender, below lower threshold acts as receiver. Best for variable workloads. Real system: Condor.

Three zones: [0..Lower]=Receiver | [Lower..Upper]=Satisfied | [Upper..Max]=Sender
Summary: Sender → heavy load. Receiver → light load. Symmetric → variable load.
